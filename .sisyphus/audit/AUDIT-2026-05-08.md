## 1. Threat Model & Goals (As Required For The Interfold)

For the audit to be actionable against the stated end-goal, we record the security objectives the artifact *must* meet.

### 1.1 Adversary

- **Computational**: PPT, classical (post-quantum is out-of-scope for v1 but should not be precluded by primitive choice).
- **Network**: Active, can reorder/drop messages, can compromise up to `t-1` of `n` parties.
- **On-chain**: Public, permissionless verifier; adversary controls the prover and may collude with a quorum below threshold.
- **Side-channels**: Out-of-scope for v1, but constant-time discipline must not be regressed.

### 1.2 Properties required (Interfold core component)

1. **DKG correctness**: With ≥`t` honest parties, the DKG outputs a valid threshold key `pk` and shares `sk_i` consistent with `pk`.
2. **DKG secrecy**: Adversary corrupting <`t` parties learns nothing about `sk` beyond what `pk` reveals (semantic security under standard lattice assumptions).
3. **Verifiable share well-formedness**: Each `sk_i` envelope is publicly verifiable as a correct Shamir share of a secret encrypted under `pk` with bounded-norm noise. Soundness ≥2⁻¹²⁸; zero-knowledge against PPT verifier.
4. **Threshold decryption correctness**: Aggregating ≥`t` honest partial decryptions of any ciphertext `c` yields `Dec(sk, c)`.
5. **Threshold decryption soundness**: A public verifier accepts an aggregated decryption ↔ ≥`t` honest partials were combined correctly. Soundness ≥2⁻¹²⁸.
6. **On-chain verifiability with O(polylog n) verifier cost**: Aggregated proof verifies on EVM in `O(polylog n)` gas in `n`.
7. **Session binding**: Every accepted proof is bound to `(session_id, epoch, ciphertext)` and consumed atomically; replay impossible.
8. **Liveness**: Honest quorum can always produce an accepted proof in bounded time.

### 1.3 Mapping to current artifact

| Property | Status | Blocker findings |
|---|---|---|
| DKG correctness        | Partial (FHE backend ok) | F23 (DKG broken at reshare) |
| DKG secrecy            | **Violated**             | F23, F20, F27, F28, F59 |
| Share WF (verifiable)  | **Violated**             | F4, F5, F8, F15, F19 |
| Decrypt correctness    | Partial (FHE backend ok) | — |
| Decrypt soundness      | **Violated**             | F8, F9, F44, F48, F54, F55, F57, F58 |
| On-chain O(polylog n)  | **Unverified**           | F9, F10 (compiler-broken), F47–F50 |
| Session binding        | **Violated**             | F9, F11 |
| Liveness               | Demo-only                | F38, F41 |

**Conclusion**: 7 of 8 required properties are violated by at least one CRITICAL finding; the 8th (decrypt correctness) is satisfied only because plaintext recovery short-circuits the proof system entirely (F57 — itself a CRITICAL).

### 2.1 Cyclo folding (F1, F2, F3, F42, F43)

#### F1 · CRITICAL · `crates/pvthfhe-cyclo/src/fold.rs:15-21` — Witness norm is `bytes.iter().max()`

The "norm estimate" used in the fold relation reads:

```rust
fn witness_norm_estimate(bytes: &[u8]) -> u64 {
    bytes.iter().copied().max().unwrap_or(0) as u64
}
```

This is the maximum byte value of the *serialized* witness, not the ∞-norm of the polynomial coefficients it represents. A witness with coefficients `[2^60, 2^60, ...]` can serialize to bytes `[0x01, 0x01, ...]` and pass any β-bound the checker enforces. The actual ∞-norm primitive in `range_check.rs` is correct but unreachable from the fold path.

**Exploit**: Forge a fold instance whose true coefficient ∞-norm is arbitrary while the byte-max stays under β. Combined with F2, this collapses fold soundness to ≈0.

**Remediation**: §R2.1 (rebuild Cyclo on real `Poly` types with coefficient-domain ∞-norm).

#### F2 · CRITICAL · `crates/pvthfhe-cyclo/src/fold.rs:37-41` — Fold challenge has 1.58 bits of entropy

```rust
let h = transcript.challenge_bytes(b"cyclo-fold");
let r = (h[0] % 3) as i64 - 1; // r ∈ {-1, 0, 1}
```

A single ternary challenge per fold round provides log₂(3) ≈ 1.58 bits of soundness. With `T = 10` folding rounds, total soundness is at most 3⁻¹⁰ ≈ 2⁻¹⁵·⁸ — i.e. forgery in ~58k attempts.

**Exploit**: Birthday-grind the transcript: try ~10⁵ fold transcripts; one will satisfy a fabricated relation.

**Reference**: Cyclo (ePrint 2026/359) requires challenges from a large subset of `R_q` with ≥ 2⁻¹²⁸ collision probability per round (Theorem 3 / Lemma 9).

**Remediation**: §R2.2 (challenges sampled from large subring; soundness budget computed and asserted).

#### F3 · CRITICAL · `crates/pvthfhe-cyclo/src/ccs_encode.rs:55-68` — `check_satisfiability` is a SHA-256 binding tautology

```rust
pub fn check_satisfiability(inst: &CcsInstance, wit: &CcsWitness) -> bool {
    let h = sha256(&[inst.bytes(), wit.bytes()].concat());
    h == inst.binding_hash
}
```

This checks that the witness was the one used to produce `binding_hash` — not that any CCS relation holds. Any `(inst, wit)` pair where `inst.binding_hash == sha256(inst ‖ wit)` passes; the prover supplies both.

**Exploit**: Set `wit := arbitrary`, compute `binding_hash := sha256(inst ‖ wit)`, ship.

**Remediation**: §R2.3 (real CCS encoder + matrix-vector check).

#### F42 · CRITICAL · `crates/pvthfhe-cyclo/src/extension.rs:56-66, 68-72` — T2 extension is byte XOR/concat

The T2 polynomial-extension test "merge" uses byte-level XOR/concat and validates norm via `bytes.iter().max()`. The construction is not a polynomial extension in any algebraic sense.

**Exploit**: Same surface as F1; chained with F1/F2 destroys fold soundness.

**Remediation**: §R2.1.

#### F43 · HIGH · `crates/pvthfhe-cyclo/src/adapter.rs:9` — `StubCycloAdapter` documented as "Production implementation"

```rust
/// Production implementation of the Cyclo adapter.
pub struct StubCycloAdapter { ... }
```

The struct is the only `CycloAdapter` impl in the workspace; `pvthfhe-aggregator` wires it as production at `folding/mod.rs:449-451`. Doc-comment misrepresentation creates a false-readiness signal for downstream reviewers.

**Remediation**: §R0.2 (label discipline) + §R2 (real implementation).

### 2.2 PVSS / NIZK well-formedness (F4, F5, F6, F18, F19, F20, F21, F22, F24, F25, F26, F27, F28)

#### F4 · CRITICAL · `crates/pvthfhe-pvss/src/nizk_share.rs:8-9, 313-329` — Proof envelope leaks witness

`ProofEnvelope` includes `witness: Vec<u8>` as a public field; verification at `:313-329` checks that the included witness hashes to `inst.commitment`. The proof *is* the witness. Zero-knowledge is violated; verifiability is reduced to a hash preimage check.

**Exploit**: Anyone who sees a transcript recovers the share secret directly.

**Remediation**: §R3.1 (real lattice NIZK; witness never leaves prover; ZK via standard simulation).

#### F5 · CRITICAL · `crates/pvthfhe-pvss/src/nizk_decrypt.rs:215-226` — Secret share derived from public statement

```rust
fn derive_secret_share(stmt: &DecryptStatement) -> u64 {
    let h = sha256(stmt.public_bytes());
    u64::from_le_bytes(h[..8].try_into().unwrap())
}
```

The "secret" used in NIZK-decrypt is a deterministic function of the public statement. Any verifier can recompute it.

**Exploit**: Any party can produce valid partial decryptions for any session without holding a share.

**Remediation**: §R3.2 (real partial-decryption NIZK with witness = `sk_i`; FS-bound `(session_id, ct)`).

#### F6 · HIGH · `crates/pvthfhe-pvss/src/encrypt.rs` — GF(256) Shamir + ad-hoc `derive_seed`

Shamir secret sharing operates over GF(256) byte-by-byte. With `t` shares and one byte-position, the per-byte forgery probability is 2⁻⁸; over a 32-byte secret with quorum-size attacks, soundness collapses well below 2⁻¹²⁸.

**Remediation**: §R1.2 (Shamir over a large prime field, e.g. BLS12-381 scalar or BN254 scalar; or RLWE-secret sharing à la SPDZ-FHE).

#### F18 · HIGH · *workspace-wide* — No `zeroize`/`secrecy` for secret material

Secret keys, shares, and noise polynomials are stored in plain `Vec<u8>` / `Poly`. No `Drop` zeroization. Memory dumps, swap, and `mem::replace` attacks recover key material trivially.

**Remediation**: §R0.3 (introduce `secrecy::Secret<T>` + `zeroize::ZeroizeOnDrop` newtype wrappers; CI lint forbids raw `Vec<u8>` for secret types).

#### F19 · HIGH · `crates/pvthfhe-nizk/src/{adapter.rs:41,294, ajtai.rs:179-183}` — Ajtai matrix from `seed_from_u64(ccs_instance_id)`

The Ajtai commitment matrix `A` is regenerated via `ChaCha20Rng::from_seed(ccs_instance_id)`. The instance ID is public; therefore `A` is a deterministic function of public data. Worse, anyone who can influence `ccs_instance_id` (the prover, in this protocol) can grind to a *trapdoored* `A` for which short preimages are easy to find.

**Exploit**: Adversarial prover grinds `ccs_instance_id` until `A` admits a known short trapdoor; binds that trapdoor to forge any well-formedness statement.

**Remediation**: §R3.3 (CRS-bound `A` from a one-time trusted setup or transparent hash-to-lattice with public input that the prover cannot influence; e.g. `H(epoch || protocol_constants)` where `epoch` is set by the chain).

#### F20 · CRITICAL · `crates/pvthfhe-pvss/src/encrypt.rs:128, 269, 288-304` — Encryption randomness deterministic in secret

```rust
let seed = derive_seed(&secret, &session_id);
let mut rng = ChaCha20Rng::from_seed(seed);
let r = sample_noise(&mut rng);
```

BFV encryption randomness `r` is a deterministic function of the secret. Two encryptions of the same secret under the same session yield identical ciphertexts; chosen-message queries trivially distinguish. Combined with F27 (GF(256) Shamir) and F28 (envelope construction), any single envelope reconstructs the secret in O(2⁸) per byte.

**Exploit**: Capture one envelope. For each byte position, enumerate 256 candidates; the unique candidate that re-derives `seed` and produces the observed ciphertext is the secret byte. Total recovery cost: 256 × 32 = 8192 BFV encryptions.

**Remediation**: §R3.4 (encryption RNG seeded from `OsRng`; smudging noise σ_smudge added per Asharov-Jain-Lopez-Alt-Tromer-Vaikuntanathan-Wichs).

#### F21 · HIGH · *workspace-wide* — No production smudging σ_smudge

Smudging noise is added only in `pvthfhe-bench/src/worked_example.rs`. Production partial-decryption paths do not add σ_smudge, leaking `sk_i` over multiple decryptions (standard threshold-FHE attack).

**Remediation**: §R3.5 (σ_smudge bound derived from circuit depth budget; added in `pvthfhe-fhe::partial_decrypt`).

#### F22 · MEDIUM · `crates/pvthfhe-cli/src/demo_nizk.rs:6, 19` — `StdRng::seed_from_u64(party_id ^ 0xD2C1_0A11)`

Demo seeds the NIZK RNG from `party_id`. With `n ≤ 255`, the seed space is 255 values. Any observer enumerates all NIZK proofs.

**Remediation**: §R3.6 (all RNGs go through `OsRng`; demo mode uses an explicit `--insecure-seed` flag that the build-time tripwire forbids in release).

#### F23 · CRITICAL · `crates/pvthfhe-fhe/src/fhers.rs:256-258` — Production threshold-Shamir reshare seeded from `party_id`

```rust
let mut rng = ChaCha8Rng::seed_from_u64(party_id as u64);
let coeffs = sample_reshare_coeffs(&mut rng);
```

This is the **production** reshare path used during DKG. Seed space is `n ≤ 255`, so all reshare coefficients (and therefore the resulting Shamir polynomial) are public. Any observer enumerates all 255 candidate seeds, recomputes coefficients, and recovers every share.

**Exploit**: Run DKG once. From any two ciphertexts, enumerate `party_id ∈ [0, 255]`, recompute the reshare polynomial, recover `sk` in O(n) work.

**Status**: **Total DKG break**. This single finding eliminates DKG secrecy regardless of all other layers.

**Remediation**: §R1.1 (DKG rebuilt on `OsRng` with explicit `Secret<Poly>` typed entropy).

#### F27 · CRITICAL · `crates/pvthfhe-pvss/src/encrypt.rs:268-285, 321-327` — Shamir over GF(256) + biased coefficient sampling

`next_nonzero_byte` rejects 0, biasing coefficient distribution; the field is GF(256). Soundness per byte is ≤ 2⁻⁸, and the bias means even fewer candidate polynomials need to be tried.

**Exploit**: With one envelope and the `next_nonzero_byte` bias, recover each byte in <128 trials.

**Remediation**: §R1.2.

#### F28 · HIGH · `crates/pvthfhe-pvss/src/encrypt.rs:391, 195-244` — Compound: any single envelope reconstructs secret

F4 + F20 + F27 compound: `ProofEnvelope` carries the witness (F4); encryption is deterministic in secret (F20); Shamir is over GF(256) with biased sampling (F27). Each is independently CRITICAL/HIGH; their composition means a single observed envelope yields the underlying secret.

**Remediation**: addressed transitively by §R1.2 + §R3.1 + §R3.4.

#### F24 · MEDIUM · `crates/pvthfhe-nizk/src/ajtai.rs:234` — TODO non-CT comparison

```rust
// TODO: switch to subtle::ConstantTimeEq
if a == b { ... }
```

Side-channel risk on share comparison.

**Remediation**: §R0.3.

#### F25 · LOW-MED · *workspace-wide* — Domain-separator constants scattered

#### F44 · CRITICAL · `crates/pvthfhe-aggregator/src/folding/mod.rs:181-228 RealFoldingScheme` — Folding is a SHA chain

```rust
fn fold(&mut self, inst: &Instance) -> Result<()> {
    self.acc = sha256(&[self.acc.as_slice(), inst.bytes()].concat()).into();

fn verify(&self, ...) -> bool {
```

**Remediation**: §R4.1 (real folding scheme: LatticeFold+ for the long-term path; Nova with a *real* step circuit for the substitution path).

#### F48 · CRITICAL · `crates/pvthfhe-compressor/src/nova/mod.rs:36-64 ToyStepCircuit` — Step circuit is `z + ext`, state_len=1, IVC_STEPS=4

```rust

    fn synthesize(&self, cs, z, ext) -> z + ext { ... }
```

**Exploit**: Genuine Nova IVC produces a valid proof — of `acc = z₀ + Σ ext_i`. The verifier accepts. The verifier does not learn anything about share well-formedness, fold correctness, or decryption.

**Remediation**: §R5.2 (real step circuit encoding the Cyclo fold relation: `(C_in, C_out, β, witness_commitment) → next_state`).

### 2.6 Noir circuits (F8, F14, F15)

#### F8 · CRITICAL · `circuits/aggregator_final/src/main.nr:9-19, 21-30` & `circuits/decrypt_share/src/main.nr:8-18, 20-30` — `rolling_digest` is non-CRH

Both circuits compute:

```noir
fn rolling_digest(items: [Field; N]) -> Field {
    let mut acc = 0;
    for i in 0..N { acc = acc + items[i] * 31; }
    acc
}
```

This is an affine combination, not a collision-resistant hash. Trivial collisions: `(a, b)` and `(a + 31, b - 1)` both digest to `31·a + 31·b` (since 31·(a+31) + 31·(b-1) = 31a + 961 + 31b - 31 = 31a + 31b + 930, *not* equal — but other linear collisions exist, and the construction has no preimage resistance regardless).

More fundamentally: even if the digest were a real hash, the circuit only enforces `pub_digest == rolling_digest(witness_items)`. There is no constraint that `witness_items` correspond to valid shares, ciphertexts, or fold instances.

**Exploit**: Choose any `witness_items` that hash to a target digest; the circuit accepts.

**Remediation**: §R7.1 (replace `rolling_digest` with `Poseidon` over a fixed domain-separated tag; encode the *protocol relation* — share WF, fold step, decrypt step — not just a digest equality).

#### F14 · MEDIUM-HIGH · `circuits/aggregator_final:103`, `circuits/decrypt_share:170` — Prover-supplied `q` unbounded

The modulus `q` is a prover-supplied input; circuits do not assert `q == ProtocolConstants::Q`.

**Exploit**: Prover supplies a degenerate `q` (e.g. small composite) for which the well-formedness checks become trivial.

**Remediation**: §R7.2 (assert `q == constant` from `circuits/check_modulus`; share `protocol_constants.nr` across circuits).

#### F15 · MEDIUM · `circuits/share_wf/src/main.nr:1` — Self-labeled NIZK surrogate

Doc-comment: *"Surrogate; real lattice NIZK happens off-chain."* Production builds nonetheless include this circuit in the aggregation pipeline as if it were a real share-WF proof.

**Remediation**: §R7.3 (delete circuit and ship a real on-chain proof of NIZK validity; or formally prove the surrogate's soundness against a precise statement).

### 2.7 End-to-end pipeline binding (F56, F57, F59)

#### F56 · CRITICAL · `crates/pvthfhe-cli/src/full_pipeline.rs:291-314 build_fold_instances` — Fold runs on synthetic constants

```rust

    FoldInstance {
        ajtai_commitment_bytes: ct_hash.to_vec(),       // 32 B from ciphertext hash
```

**Exploit**: The fold's output binds to `ct_hash` and nothing else. A prover with any valid `ct_hash` produces a valid fold transcript without holding any share.

#### F57 · CRITICAL · `crates/pvthfhe-cli/src/full_pipeline.rs:78, 234, 289` — Plaintext recovery via FHE backend bypasses proof system

**Status**: The proof system is decorative. Plaintext recovery succeeds whether or not any proof verifies.

**Remediation**: §R8.2 (aggregator refuses to release plaintext until the fold-compressed proof verifies against the same `(session_id, ct, partials)` tuple; single atomic API).

#### F59 · HIGH · `crates/pvthfhe-cli/src/full_pipeline.rs:101, 178, 236, 190` — All RNGs derived from `cfg.seed ⊕ party_id`

NIZK randomness, encryption randomness, partial-decrypt randomness, and fold randomness are all seeded from `cfg.seed` XOR'd with `party_id`. A verifier with `cfg.seed` reproduces every "private" value.

### 2.10 Appendix-crate findings (post-publication addendum)

These six findings were identified by direct read of `pvthfhe-keygen`, `pvthfhe-enclave-adapter`, `pvthfhe-api`, and `pvthfhe-core` after the initial 59-finding catalogue was written.

#### F60 · CRITICAL · `crates/pvthfhe-keygen/src/hermine.rs:30, 73-84, 292-301` — Production PVSS coefficients are deterministic SHA derivations of public inputs

`HermineAdapter` is the `KeygenAdapter` impl wired into the demo/e2e path (the module doc says it is "DEPRECATED legacy path" *and* simultaneously "real cryptography"; no `LatticePvssBfvAdapter` exists in the workspace). All polynomial coefficients — including the secret `s` (constant term) — are derived via:

```rust
let secret = derive_field_elem(&session.session_id, dealer_id, b"secret", 0);
// derive_field_elem hashes (session_id ‖ dealer_id ‖ tag ‖ index) → field elem
```

`session_id` and `dealer_id` are public. Therefore the secret, the polynomial coefficients, and every share are public functions of public data. There is no secrecy.

**Exploit**: Observe `(session_id, dealer_id)` (both broadcast). Recompute `derive_field_elem(session_id, dealer_id, b"secret", 0)`. Output is the secret.

**Status**: **Total break of the production `KeygenAdapter` advertised as "real PVSS"**.

**Remediation**: §R1 entirety — `HermineAdapter` must be replaced or its derivation must use private dealer randomness sampled from `OsRng` (the field/threshold structure is fine; the entropy is the bug).

#### F61 · CRITICAL · `crates/pvthfhe-keygen/src/hermine.rs:115-208 verify_share_set` — "Verification" is consistency-with-deterministic-derivation, not soundness

`verify_share_set` checks that published commitments match `commit(session_id, pid, secret_value)` and that share commitments match. Since both the secret and commitments are deterministically derived (F60), this verification confirms only that the dealer applied the deterministic procedure — not that any well-formedness property holds. There is no verifiable secret-sharing property: a dishonest dealer who follows the deterministic derivation passes verification trivially, and a verifier learns nothing they didn't already know.

**Exploit**: Equivalently, verification is `true ↔ dealer_used_the_specified_SHA_function`. No adversary advantage to compute; no security to provide.

**Remediation**: §R1.5 (real DKG with secrecy property test).

#### F62 · HIGH · `crates/pvthfhe-keygen/src/hermine.rs:33, 39-41, 292-301` — `NORM_BOUND_B_E = 16` is decorative

The constant `NORM_BOUND_B_E = 16` claims to encode "6σ for σ=3.19" (which would be ≈19.14, not 16). The helper `check_share_shortness(value: u64) -> bool { value <= 16 }` exists but is **never called** from the dealer path. Coefficients derived at `:292-301` are 60-bit field elements (mod `2^61 - 1`) without any norm enforcement. The lattice-norm framing in the module doc-comment is decorative.

**Exploit**: Norm-aware downstream consumers (e.g. a future R2.1 fold) would accept arbitrary 60-bit "shares" thinking they were norm-bounded.

**Remediation**: §R1.2 + §R3.0 (rebuild on a real lattice/field with explicit norm enforcement at sample time).

#### F63 · CRITICAL · `crates/pvthfhe-keygen/src/hermine.rs:376-419 reconstruct_bfv_key` — `BFVPublicKey` is `secret.to_be_bytes()`

```rust
fn reconstruct_bfv_key(&self, shares: &[Share]) -> Result<BFVPublicKey, KeygenError> {
    ...
    let secret = lagrange_interpolate(&points);
    Ok(BFVPublicKey {
        bytes: secret.to_be_bytes().to_vec(),  // 8 bytes
    })
}
```

The "reconstructed BFV public key" is an 8-byte big-endian encoding of the Lagrange-interpolated u64 secret. This is not a BFV public key — it does not have the structure `(a, b = -(a·s + e))` or any RLWE-key shape. Any caller who treats this as a BFV public key (the unit test at `:495-497` does: `assert_eq!(key.bytes.len(), 8);`) is operating on a fiction.

**Exploit**: The aggregator and any downstream encryption path that uses `reconstruct_bfv_key` is encrypting under a non-key. There is no BFV; there is no FHE.

**Status**: **Production keygen output is fictional**.

**Remediation**: §R1.5 + §R8 (real DKG outputs a real BFV public key compatible with `gnosisguild/fhe.rs`; `reconstruct_bfv_key` returns the public key the BFV library accepts).

#### F64 · HIGH · `crates/pvthfhe-enclave-adapter/src/lib.rs:112-114` — `verify_proof` unconditionally accepts

```rust
fn verify_proof(&self, _proof: &EnclaveProof, _public_inputs: &[u8]) -> Result<bool, String> {
    Ok(true)
}
```

The TEE/enclave attestation verification path returns `Ok(true)` for any proof. The enclave-adapter is the integration boundary for verifying that a remote ciphernode is running inside a genuine TEE; with this implementation, no attestation is enforced.

**Exploit**: Any non-enclave actor presents itself as a valid ciphernode. The "decentralized confidential compute" claim collapses.

**Remediation**: §R6.4 + new §R10 (real enclave attestation — Intel SGX / AMD SEV-SNP / AWS Nitro — verified against a quote-validation library; trust roots committed on-chain).

#### F65 · HIGH · `crates/pvthfhe-enclave-adapter/src/lib.rs:3` — `#![allow(missing_docs, dead_code)]`

Crate-wide allow for `missing_docs` and `dead_code`. AGENTS.md and the audit's R0 surface forbid silently masked dead code.

**Remediation**: §R0.3 extension — CI lint forbids crate-wide `#![allow(...)]` outside a documented exception list.

#### F66 · LOW-MED · `crates/pvthfhe-api/src/lib.rs`, `crates/pvthfhe-core/src/lib.rs` — Skeleton crates

`pvthfhe-api` is 6 lines (placeholder test only). `pvthfhe-core` is 9 lines (re-exports `pvthfhe-compressor`). Both crates are referenced from `Cargo.toml` workspace members but provide no API surface. No doc-comment warns the reader.

#### F70 · LOW-MED · R0.3 GATE follow-up — `pvthfhe-types` lint scope is incomplete

The R0.3 GATE lint test (`crates/pvthfhe-types/tests/secret_types_present.rs`) only flags `pub` fields of type `Vec<u8>` / `Poly` inside structs whose names match `*Secret*`, `*Share*`, or `*Sk*`. It misses two classes of real secrets:
1. **Private (non-`pub`) fields** holding `Vec<u8>` / `Poly` secret material — invisible to the lint.
2. **Secrets in structs whose names do not match the allowlist patterns** (e.g. a struct named `KeygenState` that internally holds an unwrapped `Vec<u8>` secret).

This was raised by oracle session `ses_1fbc91f4ffepBym15OcR9L7LA` while adjudicating R0.3 GREEN Strategy 4. **Remediation**: deferred to R8 GATE. Add a follow-up lint that uses `cargo expand` + `syn` visitor to flag any struct field whose type resolves to `Vec<u8>` / `Poly` AND whose containing struct is reachable from any public function returning a value annotated `#[secret]` (or marked `Secret<…>` in the new types crate). Track as a hardening item, not a launch blocker.

## 4. Methodology

### 4.1 What was inspected

- **Direct read** (auditor): all files cited in §2 by `path:line`.
- **Agent-mediated read** (4 parallel `explore` agents): Solidity contracts + Foundry tests, all Noir circuits, RNG/primitives inventory, design-vs-implementation map, Rust crate inventory.
- **Cross-checks**: Noir↔Solidity public-input ordering; `Prover.toml` schema; Stage 0 tripwire active; `unsafe`-free; `subtle::ConstantTimeEq` callsites.

### 4.2 What was *not* inspected (deferred)

- Full read of `pvthfhe-keygen`, `pvthfhe-enclave-adapter`, `pvthfhe-api`, `pvthfhe-core` (spot-checks only; no findings altered severities).
- Cryptographic review of `gnosisguild/fhe.rs` BFV implementation itself (assumed correct per upstream community review).
- Side-channel / timing analysis (out of scope v1).
- Formal verification of circuit synthesis (Noir → ACIR → backend); only source-level review.

These are queued for the rebuild's `R0` baseline.

### 4.3 Severity criteria

- **CRITICAL**: Concrete exploit path with file:line citation, breaking confidentiality, integrity, or soundness against a PPT adversary.
- **HIGH**: Same as CRITICAL but exploit requires non-trivial setup (e.g. specific RNG state, correlated session) or breaks a non-headline property (replay protection, role separation).
- **MEDIUM**: Defense-in-depth gap; no direct exploit but materially weakens the security argument.
- **LOW**: Correctness/clarity issue with no security impact.
- **INFO**: Reporting/documentation issue.

### 4.4 Auditor's confidence

- High confidence in CRITICAL and HIGH findings: each has a quoted code excerpt and a stated exploit.
- Medium confidence in MEDIUM/LOW findings: some are catalogue-driven (workspace-wide patterns) and may admit exceptions in deferred crates.
- The 22 CRITICAL findings are independent; addressing any subset does not eliminate the others.
