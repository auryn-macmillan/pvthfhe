## Executive Summary

188 new findings identified across 4 audit layers. The codebase has received extensive remediation (R0–R10, 144/178 plan tasks complete), but deep structural issues remain at every cryptographic layer. The **NIZK verifier is a stub** returning `Ok(())` unconditionally. The **on-chain HonkVerifier is a tautology** (`keccak256(proof) == publicInputs[0]`). The **fold verification never enforces CCS satisfiability**. The **compressor step circuit proves field addition, not folding**. These are not subtle cryptographic weaknesses — they are complete bypasses.

## Consolidated Findings by Severity

### CRITICAL (21 findings)

#### C1 — NIZK verification is a complete stub (Lattice Layer)
- **File**: `crates/pvthfhe-pvss/src/nizk_share.rs:214-220`
- `verify_d2_hash_binding()` unconditionally returns `Ok(())`. Called during verification at line 182. The D2 hash binding check — a core component of the NIZK — is completely bypassed. Any proof passes.

#### C2 — Proof-encoded session_id/participant_id discarded during verification
- **File**: `crates/pvthfhe-nizk/src/adapter.rs:165-167`
- The verifier parses `session_id` and `participant_id` from the proof, then explicitly discards them with `let _ = ...`. Only the pvss_commitment is checked. An attacker can embed arbitrary session/participant values.

#### C3 — On-chain HonkVerifier is a tautology
- **File**: `contracts/src/generated/HonkVerifier.sol:16-18`
- `verify()` returns `keccak256(proof) == publicInputs[0]`. Any caller sets `publicInputs[0] = keccak256(proof)` and passes all verification.

#### C4 — All 6 commitment checks bypassed in on-chain verifier
- **File**: `contracts/src/PvtFheVerifier.sol:146-156`
- `verify()` builds a 7-element `publicInputs` array (ciphertextHash, plaintextHash, aggregatePkHash, dkgRoot, epoch, participantSetHash, dCommitment). HonkVerifier only checks `publicInputs[0]`. All commitment hashes are structurally ignored.

#### C5 — Attestation signature never verified
- **File**: `contracts/src/PvtFheVerifier.sol:195-218`
- `verifyWithAttestation()` checks `attestors[attestation.signer]` but never verifies the `attestation.signature` bytes. No `ecrecover`, no EIP-712. The signature field is dead data.

#### C6 — Decrypt shares are completely unlinked to NIZK proofs
- **File**: `crates/pvthfhe-cli/src/full_pipeline.rs:113-127` vs `:256-269`
- The pipeline proves NIZK properties on synthetic `demo_nizk` witnesses, but partial decryption uses real BFV secret keys via `backend.partial_decrypt()`. The NIZK proofs and actual decryption shares are independent data flows. The compressed proof verifies properties of data that was never used.

#### C7 — Demo NIZK secret share is deterministically derived from public data
- **File**: `crates/pvthfhe-cli/src/demo_nizk.rs:70-77`
- `demo_secret_share()` computes `SHA256(session_id || participant_id || pk_bytes) % 65537`. All inputs public. Zero entropy. The entire NIZK layer proves properties of a publicly determinable value.

#### C8 — CCS satisfiability never enforced in any verification path
- **File**: `crates/pvthfhe-cyclo/src/fold.rs:127-192`
- `verify_fold` recomputes the fold chain and checks commitment/public_io match, but never calls `check_satisfiability`. Instances can pass fold verification while violating the CCS relation `M·z ⊙ z == 0`.

#### C9 — CCS satisfiability falls back to SHA-256 tautology when matrix empty
- **File**: `crates/pvthfhe-cyclo/src/ccs_encode.rs:130-133`
- When `ccs_matrix` is empty (the default: `ccs_encode.rs:54`), `check_satisfiability` falls back to `check_sha_binding` which verifies `SHA256(ajtai_hash || public_io_hash || witness_bytes) == sha256_binding` — a tautology since the binding was computed identically.

#### C10 — CycloFoldStepCircuit proves field addition, not folding
- **File**: `crates/pvthfhe-compressor/src/nova/mod.rs:107-120`
- `CycloFoldStepCircuit` generates step constraints `z_i[0] + external, z_i[1] + external, z_i[2] + 1`. The entire Nova IVC chain proves that field addition was performed, not that cryptographic folding occurred.

#### C11 — Secret key material stored without Zeroize or Secret<T>
- **File**: `crates/pvthfhe-fhe/src/fhers.rs:39-48`
- `PartyState` stores `sk_poly_sum: Vec<i64>` (full secret-key coefficients) and `sk_shamir_shares: Vec<Vec<i64>>` — both plain `Vec` without `Secret<T>`, `Zeroize`, or `ZeroizeOnDrop`. The struct derives `Clone + Debug` enabling secret key duplication and debug-printing.

#### C12 — Fiat-Shamir domain separator uses lossy UTF-8 encoding
- **File**: `crates/pvthfhe-nizk/src/fiat_shamir.rs:47-48`
- `Transcript::new` uses `String::from_utf8_lossy(session_id)`. Two different byte sequences that map to the same lossy UTF-8 output produce identical domain separators — non-injective encoding enables collisions.

#### C13 — Fiat-Shamir challenge derived from statement only, not binding prover
- **File**: `crates/pvthfhe-pvss/src/nizk_share.rs:413-428`
- `derive_challenge` computes the challenge from statement fields only (session_id, pk, ciphertext, share_commitment). No prover commitment is absorbed before the challenge. Violates Fiat-Shamir transform: challenge doesn't bind the prover.

#### C14 — Hermine adapter derives ALL polynomial coefficients from public SHA
- **File**: `crates/pvthfhe-keygen/src/hermine.rs:292-301` (F60 from prior audit)
- `generate_shares` derives secret + all random coefficients from `(session_id, dealer_id, tag, index)` via SHA-256. Any observer with session_id+dealer_id can reconstruct all shares.

#### C15 — Encryption randomness deterministically derived from secret+public data
- **File**: `crates/pvthfhe-pvss/src/encrypt.rs:493-506`
- `derive_share_randomness` derives all encryption randomness from `(session_id, recipient_index, recipient_pk, secret)`. Deterministic from inputs, not from a CSPRNG.

#### C16 — Decrypt NIZK witness deterministically forged from loop index
- **File**: `crates/pvthfhe-cli/src/pvss_support.rs:82-84`
- `secret_key_bytes = vec![index + 1; 64]` and `decryption_noise = ... ^ 0x5A; 64`. The "secret key" and "noise" are completely deterministic from the loop counter.

#### C17 — Toy witness test is circular (self-fulfilling)
- **File**: `contracts/test/P3RealVerifierBase.t.sol:26-28`
- `validProof = abi.encodePacked("pvthfhe-valid-proof")` and `validPublicInputs[0] = keccak256(validProof)`. This satisfies the HonkVerifier tautology exactly. All "honest proof verifies" tests are circular.

#### C18 — Legacy fold path `RealFoldingScheme` is SHA-256 hash chain
- **File**: `crates/pvthfhe-aggregator/src/folding/mod.rs:118-192`
- `finalize()` produces `hash_parts(&[Tag::Finalize, encode_accumulator(acc)])` — a 32-byte SHA-256 hash, not a real cryptographic proof. `verify_acc` never calls CCS satisfiability.

#### C19 — `CycloFoldingAdapter` wraps `LegacyHashChainAdapter` — name fraud
- **File**: `crates/pvthfhe-aggregator/src/folding/mod.rs:475-477`
- Constructor creates `inner: LegacyHashChainAdapter`. Despite the "Cyclo" name, the actual implementation is the legacy hash-chain adapter. The name falsely signals real LatticeFold+ is active.

#### C20 — Decrypt NIZK check is `nizk[0] != 1` — trivially bypassed
- **File**: `crates/pvthfhe-aggregator/src/decrypt/mod.rs:53,80`
- Default `nizk = vec![1]`, validation checks `payload.nizk.is_empty() || payload.nizk[0] != 1`. Any share with `nizk = vec![1]` passes the NIZK check unconditionally.

#### C21 — Epoch DOS via `verifyAndConsume` — adversary can burn epochs
- **File**: `contracts/src/PvtFheVerifier.sol:165-191`
- `verifyAndConsume()` marks epoch consumed BEFORE verifying the proof. If the proof is invalid, epoch is permanently consumed. Adversary can submit junk proofs to consume all valid epochs.

### HIGH (60 findings — sampled)

#### H1 — Witness norm check and CCS satisfiability use different serializations
- `fold.rs:67-69` vs `ccs_encode.rs:136`
- Norm check reads bytes as u64-LE; satisfiability reads bytes as Fr-LE. Different byte interpretations on the same data.

#### H2 — SRS is deterministically generated from epoch_hash — anyone can reproduce
- `compressor/src/nova/mod.rs:171-175`
- `ChaCha20Rng::from_seed(Keccak256(epoch_hash || "nova-srs" || "-seed"))`. Anyone knowing epoch_hash can reproduce the entire SRS including proving key.

#### H3 — Epoch IS NOT validated against proof struct — proofs lack epoch binding
- `nizk_decrypt.rs:322`
- `decode_opened_proof_body` constructs decoded statement with `epoch: 0` hardcoded, regardless of original proof content.

#### H4 — Encrypt ignores caller-supplied RNG, uses thread_rng directly
- `fhers.rs:573-574`
- `let mut rng = thread_rng();` while `_rng: &mut dyn RngCore` parameter is ignored.

#### H5 — Aggregator circuit hardcodes N_PARTICIPANTS=3, THRESHOLD=2
- `aggregator_final/src/main.nr:15-16`
- Lagrange recombination weights [3, -3, 1] valid only for participants {1,2,3}. Cannot verify arbitrary threshold decryptions.
