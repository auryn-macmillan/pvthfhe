## Findings

### H1: Ternary Challenge Mod-3 Bias (Non-Uniform Distribution) · HIGH

**File**: `crates/pvthfhe-nizk/src/sigma.rs:736-749`, `crates/pvthfhe-cyclo/src/fiat_shamir.rs`, `crates/pvthfhe-compressor/src/nova/greyhound_pcs.rs`

**Vulnerability**

Three separate challenge derivation paths use `byte % 3` or `Fr` mapped to ternary `{-1, 0, 1}`:

```text
sigma.rs: fr_to_ternary() → maps Fr to {-1,0,1} via half-modulus comparison
cyclo/fiat_shamir.rs: hash[0] % 3 → 86:85:85 bias (256 % 3 = 1)
greyhound_pcs.rs: byte % 3 from ChaCha20 output → same mod-3 bias
```

The sigma.rs path uses `Fr` reduction which avoids byte-level mod bias but introduces a field-level mapping that depends on `Fr::MODULUS` being odd. The ternary mapping is: `0 → 0`, value in upper half of field → -1, otherwise → 1. Since `|Fr|` is prime (odd), the three buckets are: `{0}` (1 element), `{1..(p-1)/2}` (exactly (p-1)/2 elements), and `{(p+1)/2..p-1}` (exactly (p-1)/2 elements). This gives probabilities: 0: 1/p, 1: (p-1)/(2p) ≈ 0.5, -1: (p-1)/(2p) ≈ 0.5. For BN254 (p ≈ 2^254), the bias is negligible (~2^-254). The cyclos/greyhound byte-level `% 3` bias at ~0.4% per challenge accumulates over multi-round repetition.

**Impact**

With 10-fold rounds and 0.4% bias per round, the effective soundness degrades from the theoretical (2/3)^k. An attacker who can influence which transcripts get challenged gains a marginal advantage. For k=90 rounds targeting 2^-142 soundness, the degradation is negligible; for k=10 (cyclo fold), the accumulated bias is measurable (~4% deviation from uniform). Severity is HIGH due to multiplicity across three subsystems.

**Fix**

Replace `byte % 3` with rejection sampling: discard bytes ≥ 252 and map remaining uniformly to {-1,0,1}. For sigma.rs `fr_to_ternary`, the current implementation is statistically near-uniform; add a comment explaining the analysis. For greyhound, use the same rejection-sampling approach.

```rust
// Replacement for byte % 3:
fn uniform_ternary(byte: u8) -> Option<i64> {
    if byte >= 252 { return None; }
    let bucket = byte / 84; // 252 / 3 = 84
    Some(match bucket { 0 => -1, 1 => 0, _ => 1 })
}
```

### H2: Mixed Endianness in D2 Hash-Bridge Commitment · HIGH

**File**: `crates/pvthfhe-nizk/src/hash_bridge.rs`

**Vulnerability**

```rust
// hash_bridge.rs — commit()
h.update(session_id.as_bytes());
h.update(participant_id.to_le_bytes());  // LITTLE-ENDIAN
h.update(secret_share.to_be_bytes());    // BIG-ENDIAN
```

The commitment function hashes `participant_id` as LE bytes and `secret_share` as BE bytes with no explicit domain separator or justification. Mixed endianness is unconventional in cryptographic commitments and creates a maintenance hazard: if any caller switches endianness, commitments become incompatible. More critically, the lack of a domain separator between fields means:

`SHA256(sid || pid_le || ss_be)` cannot distinguish `(sid="abc", pid=1, ss=42)` from `(sid="ab", pid=1, ss=42)` if `sid` ends with `\x01`.

**Impact**

The commitment scheme is vulnerable to cross-domain injection: an attacker can craft ambiguous encodings if `session_id` contains byte values that overlap with LE-encoded small integers. With current usage (session_id is hex-encoded or random bytes), exploitability is LOW, but the construction is fragile. Severity is HIGH because commitment binding is a foundational property.

**Fix**

1. Add a domain separator prefix: `h.update(b"pvthfhe-d2-hash-bridge-commit/v1")`
2. Use consistent endianness (LE throughout, or BE throughout)
3. Use length-prefixed encoding for all variable-length fields

```rust
fn commit(session_id: &[u8], participant_id: u16, secret_share: u64) -> [u8; 32] {
    let mut h = Sha256::new();
    h.update(b"pvthfhe-d2-hash-bridge/v1");
    h.update(&(session_id.len() as u32).to_be_bytes());
    h.update(session_id);
    h.update(&participant_id.to_be_bytes());
    h.update(&secret_share.to_be_bytes());
    h.finalize().into()
}
```

### H3: Non-Standard Fiat-Shamir Transcript Finalization · HIGH

**File**: `crates/pvthfhe-nizk/src/fiat_shamir.rs`

**Vulnerability**

```rust
pub fn challenge_bytes(&self, label: &[u8], output_len: usize) -> Result<Vec<u8>, NizkError> {
    let mut hasher = self.hasher.clone();
    hasher.update(&(output_len as u64).to_be_bytes());
    hasher.update(label);
    let digest = hasher.finalize();
    // ... expand via counter mode ...
}
```

The `challenge_bytes()` method clones the internal hasher, adds the challenge label, and finalizes the clone — but the original `self.hasher` is NOT reset. The doc comment states: "future absorb/challenge reflect the finalised state", but the internal hasher is not updated with the challenge label. This means:

1. Subsequent `absorb()` calls feed into a state that does NOT include the challenge label
2. The transcript does NOT properly separate challenge derivation from message absorption

This violates the standard FS model where challenge derivation absorbs a domain separator into the transcript before finalizing.

**Impact**

A prover could potentially construct two transcripts where: (a) the challenge is computed over some messages A, but (b) subsequent verifier checks use messages B ≠ A, because the transcript absorption didn't commit to the challenge label. In practice, the current code uses `absorb` before `challenge_bytes`, so the transcript IS updated with challenge-relevant data before the clone. The bug is that the label fed to `challenge_bytes` is committed via a throwaway clone, not the persistent hasher. Severity is HIGH due to non-standard transcript model.

**Fix**

Either:
1. Make `challenge_bytes` absorb the label into the persistent hasher before cloning (standard model), or
2. Document clearly that the transcript model is non-standard and prove it's equivalent

```rust
pub fn challenge_bytes(&mut self, label: &[u8], output_len: usize) -> Result<Vec<u8>, NizkError> {
    // Absorb into persistent state first
    self.absorb(b"challenge-label", label)?;
    // Now clone for finalization
    let mut hasher = self.hasher.clone();
    hasher.update(&(output_len as u64).to_be_bytes());
    // ...
}
```

### H4: ~~Final Decryption Not Verified Against Group Public Key~~ → Downgraded to LOW (Mitigated) · LOW

**Files**: `crates/pvthfhe-fhe/src/fhers.rs:1456-1473`

**Original Finding** (from prior audit): Native `aggregate_decrypt` returns plaintext without re-encrypt-and-verify check.

**Re-analysis (2026-06-05)**: The production verification pipeline already covers this:

```
partial_decrypt_with_witness (per-party NIZK)
  → aggregator_final Noir circuit (C7: Lagrange recombination + S-Z check)
  → IVC proof (Nova step circuit chain)
  → PvtFheVerifier.sol on-chain (Honk → Sumcheck → Shplemini)
```

The C7 circuit (`circuits/aggregator_final/src/main.nr` lines 20-24) proves:
1. `sum(lambda_i) = 1` (Lagrange identity)
2. `sum(lambda_i * d_i(r)) = pt(r)` (Schwarz-Zippel recombination)
3. `plaintext_commitment = Poseidon(pt)` (binding)

The native `aggregate_decrypt` without post-verification is safe because:
- In production: it's **never** the final step — the Noir proof + IVC + on-chain verifier always follows
- In tests/simulators: no malicious adversary is modeled
- The NIZK proof for each share is already verified before aggregation reaches `decrypt_from_shares`

**Severity downgrade** (HIGH → LOW): No code fix needed. The existing NIZK + Noir + IVC pipeline provides correctness verification. This is a **documentation gap** — the native `aggregate_decrypt` method should carry a doc comment noting it's irreducible (non-verifiable without the full pipeline).

**Resolution**: Documentation-only. Add method doc to `aggregate_decrypt` explaining the split between native (test/simulator) and verified (NIZK+Noir+IVC) paths.

### H5: HermineAdapter Direct Usage Bypasses Feature Gate · HIGH

**File**: `crates/pvthfhe-keygen/tests/honest_run.rs:3`, `crates/pvthfhe-keygen/tests/adversarial.rs:4`, `crates/pvthfhe-keygen/tests/forged_share_rejection.rs:1`, `crates/pvthfhe-keygen/tests/hermine_pvss_adversarial.rs:3`, `crates/pvthfhe-bench/src/bin/bench_p4.rs:8`

**Vulnerability**

The `HermineAdapter` is properly feature-gated with `#[cfg(feature = "hermine")] compile_error!()` in its module declaration. However, test files and benchmark binaries import and construct `HermineAdapter` DIRECTLY without going through the feature gate:

```rust
// honest_run.rs:3 (test file — compiles without hermine feature!)
use pvthfhe_keygen::{hermine::HermineAdapter, KeygenAdapter, Participant};
```

This works because `#[cfg(feature = "hermine")]` only prevents compilation of the *module body* when the feature is disabled — but the module is still declared and the type is defined. The `compile_error!()` fires only at the `mod hermine;` declaration, not at `use` sites. However, test crates re-export the module path, so tests can still import `HermineAdapter` (it compiles as a unit struct with no methods when the feature is disabled, which still passes type checking but can't do anything).

The actual impact: tests using `HermineAdapter` compile without the `hermine` feature because `#[deprecated]` doesn't prevent compilation, and the unit tests are NOT gated behind `#[cfg(feature = "hermine")]`. When built without the feature, these tests exercise a no-op struct.

**Impact**

1. Tests that should test real PVSS logic test nothing (silent no-op)
2. The `bench_p4.rs` binary produces meaningless benchmark numbers when compiled without the feature
3. A developer could accidentally use `HermineAdapter` thinking it works, when it's a deprecated stub

**Fix**

Gate all HermineAdapter usage behind `#[cfg(feature = "hermine")]`:

```rust
// tests/honest_run.rs
#[cfg(feature = "hermine")]
use pvthfhe_keygen::hermine::HermineAdapter;

#[cfg(feature = "hermine")]
#[test]
fn honest_run_test() { /* ... */ }

#[cfg(not(feature = "hermine"))]
#[test]
fn hermine_feature_disabled_test() {
    // Verify hermine cannot be used without the feature
}
```

### H6: TFHE Bootstrap Uses Hardcoded Re-Encryption Seeds · HIGH

**File**: `crates/pvthfhe-fhe-poulpy/src/poulpy_backend_impl/tfhe_ops.rs:248-256`

**Vulnerability**

```rust
let seed_bytes_1: [u8; 32] = [0xABu8; 32];
let seed_bytes_2: [u8; 32] = [0xCDu8; 32];
let seed_rng_1 = ChaCha8Rng::from_seed(seed_bytes_1);
let seed_rng_2 = ChaCha8Rng::from_seed(seed_bytes_2);
```

The TFHE bootstrap operation uses hardcoded seeds (`0xAB` and `0xCD` repeated 32 times) for re-encryption randomness. This means:

1. Every bootstrap operation uses the same "random" mask
2. An observer who knows the mask can recover the bootstrapped value
3. This is a simplified bootstrap — not real blind rotation — but the hardcoded seeds violate the cryptographic requirement for fresh randomness

**Impact**

If the Poulpy TFHE backend is used in any real deployment path, bootstrap outputs are deterministic and all noise-based security is eliminated. Currently, Poulpy is behind `enable-lazer` (off by default) and is not the active backend, but the code path exists and could be accidentally activated.

**Fix**

Replace hardcoded seeds with `OsRng`-derived seeds:

```rust
let mut seed_bytes_1 = [0u8; 32];
let mut seed_bytes_2 = [0u8; 32];
rng.fill_bytes(&mut seed_bytes_1);
rng.fill_bytes(&mut seed_bytes_2);
```

## MEDIUM Severity Findings (8)

### M1: Poseidon Merkle Tree Lacks Leaf/Internal Domain Separation · MEDIUM

**File**: `crates/pvthfhe-compressor/src/merkle.rs`

The 8-ary Merkle tree uses `hash8_native()` (Poseidon permutation) for both leaf and internal nodes without domain separation. While Poseidon is a sponge-based construction not vulnerable to length-extension attacks (unlike SHA-2), the absence of domain separation between leaf and internal node hashing is a best-practice violation that could enable second-preimage attacks if the leaf value encoding overlaps with internal node formats.

**Fix**: Add one-bit domain separator to leaf vs internal Poseidon states.

### M2: Inline Domain Tags Not Consolidated · MEDIUM

**Files**: `sigma.rs`, `schnorr.rs`, `greyhound_pcs.rs`, `fiat_shamir.rs`, `adapter.rs`

Multiple protocol components use inline string literals for domain separation instead of referencing the centralized `pvthfhe-domain-tags` crate. This creates a maintenance risk: domain tags could inadvertently collide if two developers choose the same label string. The `domain-tags` crate already has 24 registered tags.

**Fix**: Move all inline domain strings into `domain-tags` and reference them by enum variant. Add a CI lint that forbids raw string literals in hash operations.

### M3: Noir Circuit Hardcodes N_PARTICIPANTS=3 · MEDIUM

**File**: `circuits/aggregator_final/src/main.nr:15-16`

```noir
let LAGRANGE_WEIGHTS: [Field; 3] = [3, -3, 1];
```

Lagrange recombination weights are valid only for participants {1,2,3}. Cannot verify threshold decryptions for arbitrary n. This was identified in the prior audit (H5) and documented but not yet parameterized.

### M5: Greyhound PCS Challenge Uses ChaCha20 with Fixed Domain Seed · MEDIUM

**File**: `crates/pvthfhe-compressor/src/nova/greyhound_pcs.rs`

The Greyhound PCS uses `Keccak256` for matrix generation seed (`"pvthfhe-greyhound-pcs-v1"`) and challenge derivation (`"pvthfhe-greyhound-challenge-v1"`) but does not bind the session identifier into either hash. A proof produced for session A could be reused in session B.

**Fix**: Add `session_id` and `prover_id` to Greyhound challenge hashing.

### M6: FS Challenge Uses Single Mod-3 Byte per Round (Cyclo) · MEDIUM

**File**: `crates/pvthfhe-cyclo/src/fiat_shamir.rs`

The `sample_challenge()` in `CycloTernaryTranscript` uses `hash[0] % 3` — exactly one byte per challenge. With 10-fold rounds, only 10 bytes of the 32-byte hash are consumed. This is wasteful but not a security issue per se. However, combined with the mod-3 bias (H1), it means only 10 bytes determine the entire fold challenge.

**Fix**: Use all hash bytes (rejection sampling) and aggregate into a single challenge vector.

### M7: Sigma Protocol Verifier Equation Bypassable with Zero Witness · MEDIUM

**File**: `crates/pvthfhe-nizk/src/sigma.rs`

The sigma protocol verifier checks `d = c*s + e (mod Q)` for ternary s. If s = 0, then any e produces a valid-looking equation: `d = c*0 + e = e`. A prover who sets s = 0 (ternary valid!) and e = d can trivially satisfy the relation. This is caught by the ternary challenge (ch ∈ {-1,0,1}), but the check that s_i is nonzero is made in `verify_witness_norm` which only checks ∞-norm bounds, not zero rejection.

**Fix**: Add explicit `s_i != 0` check in the verifier for each round.

---

### M8: No Noir In-Circuit Verifier for BFV Sigma Proofs · MEDIUM

**File**: `crates/pvthfhe-nizk/src/bfv_sigma.rs`

BFV sigma proofs are only verified natively; there is no Noir/R1CS verifier. This means:
1. BFV sigma proofs cannot be used inside Nova step circuits
2. The Schwarz-Zippel evaluation path (`compute_sigma_sz_data`) is the workaround
3. This is documented as a limitation but creates a proof-system gap

**Fix**: Documented as deferred. Consider adding an R1CS verifier for BFV sigma in a future milestone.
