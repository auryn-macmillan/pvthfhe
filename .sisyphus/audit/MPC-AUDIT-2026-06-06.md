## Executive Summary

**12 findings** (3 CRITICAL, 4 HIGH, 3 MEDIUM, 2 LOW) across 5 domains. The 2026-06-05 audit's 19 findings are nearly all remediated (14 confirmed fixed, 2 mitigated, 2 still pending). However, fresh deep-dive analysis reveals 3 CRITICAL structural gaps that the prior audit did not raise:

1. **G-N8 (CRITICAL): N=8 circuit vs N=8192 production** — The Noir circuit that serves as the on-chain trust anchor operates on N=8 polynomials while production RLWE uses N=8192. The reduction from N=8192 to N=8 is performed in untrusted native Rust code with NO proof of correctness-preservation.
2. **S1 (CRITICAL): Dual native/in-circuit proof paths diverge** — The native sigma verifier and the C7 Noir circuit use DIFFERENT transcript constructions (Poseidon-based scalar challenge vs Merkle-based Lagrange recombination) for the same security claim. A proof that passes native verification may fail circuit verification, or vice versa, creating an uncertain security boundary.
3. **S2 (CRITICAL): No in-circuit FHE multiplication proof** — The `FheComputeStepCircuit` only proves FHE Add operations. FHE Mul (multiplication) — the entire raison d'être of FHE — has no verifiable proof in the circuit.

The codebase has excellent hygiene for an early-stage research prototype but should NOT be deployed until the CRITICAL gaps are closed.

## Fresh Findings (2026-06-06)

### G-N8: N=8 Circuit vs N=8192 Production — Proof of Correctness Gap · CRITICAL

**Vulnerability**

The Noir circuits that serve as the on-chain verifier's trust anchor operate on **N=8** polynomial coefficients, while production RLWE uses **N=8192**:

```text
circuits/aggregator_final/src/main.nr → global N: u32 = 8
circuits/decrypt_share/src/main.nr     → global N: u32 = 8
circuits/nova_state_commitment/        → nova_final_plaintext: [Field; 8]
```

The mapping from N=8192 to N=8 occurs in native Rust (`aggregate_decrypt_raw_result_poly` in `full_pipeline.rs`) which is **NOT provably correctness-preserving**. A malicious aggregator (untrusted by design) can choose N=8 projections that satisfy circuit constraints but correspond to an incorrect N=8192 plaintext.

Since the circuit IS the on-chain verifier's decision boundary, anything the circuit accepts is treated as valid. This is a **structural soundness gap**: there is no formal reduction from N=8192 correctness to N=8 verification.

**Code path**:
```
full_pipeline.rs: aggregate_decrypt_raw_result_poly()
  → projects N=8192 plaintext to N=8
  → builds C7 witness for aggregator_final circuit
  → circuit verifies: sum(lambda_i * d_i(8)) = pt(8)  ← only correct for N=8!
```

**Impact**

Any on-chain verification result may be accepting semantically incorrect decryptions. The gap is structural — no amount of sigma protocol soundness can close it, because the soundness argument itself operates in the N=8192 domain while verification occurs in N=8.

**Fix**

Either:
1. Scale Noir circuits to N=8192 (requires `generic_const_exprs` or circuit specialization)
2. Provide a formal reduction theorem: "if the Lagrange recombination holds for N=8 evaluation points, then it holds for N=8192 with probability ≥ 1 - ε" — and implement the multi-point evaluation scheme
3. OR document this as a research-prototype limitation with fail-closed behavior

### S1: Dual Native/Circuit Proof Paths Use Incompatible Transcripts · CRITICAL

**Vulnerability**

The per-share decryption correctness is checked TWICE — once by the native sigma verifier and once by the C7 Noir circuit — but they use **different transcript constructions** for what should be the same security claim:

| Layer | Proof | Transcript | Security |
|-------|-------|-----------|----------|
| Native (adapter.rs) | Sigma protocol | Poseidon-based ternary scalar challenge bound to session_id + participant_id + round_index + d_commitment | ~142 bits (k=90) |
| In-circuit (aggregator_final) | C7 Lagrange recombo | Merkle-hash based: Poseidon([share_hashes...]), Schwartz-Zippel challenge r = Poseidon(7 public inputs) | Soundness ~2^-135 |

The native path verifies: `c*z_s + z_e = t + ch*d_i` (mod Q) through the sigma protocol.
The circuit path verifies: `sum(lambda_i * d_i(r)) = pt(r)` via Lagrange recombination at a single evaluation point r.

These are **different verification equations**. A malicious prover could:
1. Produce a sigma proof that is valid for one `(c, d_i)` pair
2. Submit different `d_i` values to the circuit that satisfy the Lagrange equation

The native verifier and circuit verifier do not share a unified transcript, creating an ambiguity about which verification constitutes the "real" security check.

**Code path**:
```
adapter.rs: verify() → sigma::verify_multi() → derive_challenge_from_commitment()
                                                     (Poseidon, session+round+d_commitment)

aggregator_final/main.nr → eval_poly(pt, r) vs sum(lambda_i * eval_poly(share_i, r))
                            where r = Poseidon(7 public inputs, DOMAIN_CHALLENGE_DERIVE)
```

**Impact**

The security boundary between native verification and circuit verification is undefined. The protocol lacks a proof that (native acceptance ∧ circuit acceptance) ⇒ correct decryption, because the statements verified by each path are not equivalent.

**Fix**

1. Unify the transcript: Have the sigma proof bind to the SAME Schwartz-Zippel challenge point `r` used in the circuit
2. OR: Remove native verification entirely and rely solely on the circuit path (with N=8192 circuits)
3. OR: Prove equivalence of the two verification relations under the commitment scheme

### S2: No In-Circuit FHE Multiplication Proof · CRITICAL

**Vulnerability**

The `FheComputeStepCircuit` (`fhe_compute_circuit.rs`) only proves FHE Add operations. FHE Mul (ciphertext multiplication) — the fundamental operation that makes FHE useful — has **no verifiable proof in any circuit**:

```
fhe_compute_circuit.rs → FheComputeStepCircuit
  → in-circuit FHE coefficient addition with Merkle inclusion ✓
  → in-circuit FHE multiplication    ✗ (NOT IMPLEMENTED)
```

The circuit performs coefficient-wise addition of input ciphertexts. But FHE multiplication requires polynomial multiplication modulo (X^N+1), which produces O(N^2) coefficient interactions. Without in-circuit proof of multiplication, any FHE computation involving Mul operations is unverifiable.

**Impact**

The "verifiable FHE operations" claim in SECURITY.md and README.md is misleading. Only Add is verifiable. An untrusted compute provider performing Mul operations can return arbitrary results without cryptographic accountability.

**Fix**

1. Implement in-circuit FHE Mul verification (polynomial convolution with modulus reduction) — non-trivial, O(N^2) constraints
2. OR: Restrict the verifiable FHE claim to Add-only operations and update all documentation
3. OR: Use the Cyclo fold-based approach to fold per-Mul NIZK proofs, similar to the decrypt path

### H7: Sigma Protocol Witness Poly-Padding Allows Witness Substitution · HIGH

**File**: `crates/pvthfhe-nizk/src/adapter.rs:507-512`

**Vulnerability**

```rust
fn pad_or_truncate_to_rlwe_n(v: &[i64]) -> Vec<i64> {
    let mut out = vec![0i64; rlwe_n()];  // N=8192
    let take = v.len().min(rlwe_n());
    out[..take].copy_from_slice(&v[..take]);
    out
}
```

When the witness `secret_share_poly` is shorter than N=8192, it is zero-padded. The verifier has no way to distinguish: (a) a genuine ternary polynomial of length k with trailing zeros, from (b) a truncated witness where the prover only supplies the first k coefficients and lets the rest be zero.

The sigma protocol verifier checks `z_s = y_s + ch*s_i` element-wise. If `s_i[j] = 0` for j ≥ k, then `z_s[j] = y_s[j]` (just the mask), and the verifier accepts the response because truncation doesn't violate the ternary constraint.

A prover with a valid but short witness can create an "unpadded" proof that appears to prove a full N=8192 witness relationship. The Ajtai commitment check partially mitigates this (A·s_i = commitment), but the commitment is checked structurally (not all-zeros) rather than computationally (not a full opening check).

**Impact**

The sigma protocol can accept proofs for witnesses of incorrect length, potentially allowing a prover to reuse a shorter witness as if it were a full-length one. Combined with the N=8/N=8192 gap (G-N8), this creates a witness-substitution surface.

**Fix**

Require exactly `rlwe_n()` coefficients in the witness, rejecting both shorter and longer inputs:
```rust
fn validate_witness_length(v: &[i64]) -> Result<(), NizkError> {
    if v.len() != rlwe_n() {
        return Err(NizkError::InvalidInput("witness length must match N"));
    }
    Ok(())
}
```

### H8: Schnorr Blind Signature Nonce Determinism (Rogue-Key Risk) · HIGH

**File**: `crates/pvthfhe-nizk/src/schnorr.rs`

**Vulnerability**

The Schnorr signature module provides DKG round binding but does NOT implement commit-before-reveal or proof of possession for public keys. Per the MPC audit checklist:

> **Rogue-Key Attack: No Commit-Before-Reveal and No Proof of Knowledge**. Public-key aggregation, DKG, and VSS need either commit-before-reveal, proof of possession, or both. Otherwise an adversary chooses its key after seeing honest keys and cancels them.

The DKG protocol's Round 1 does have commit-reveal (H2 fix verified), but the Schnorr adapter used within the DKG for NIZK well-formedness does not independently bind to that commitment. An adversary could:
1. Observe honest Schnorr public keys during DKG
2. Choose a rogue key that cancels specific honest keys in the aggregate
3. The Schnorr NIZK would verify (it only checks the relation, not PoP)

**Impact**

An adaptive adversary can bias the DKG public key by choosing their contribution after observing honest contributions. The C5 proof covers aggregate key formation, but individual Schnorr NIZKs don't independently prevent rogue-key attacks.

**Fix**

Add proof-of-possession to the Schnorr module: each party must prove knowledge of `sk` relative to their declared `pk` before the key is accepted into the aggregate.

### H9: Missed Inline Domain Separators Create Potential Tag Collisions · HIGH

**Files**: Multiple (`sigma.rs`, `adapter.rs`, `fiat_shamir.rs`, `schnorr.rs`)

**Vulnerability**

Several inline domain separator strings remain outside the `pvthfhe-domain-tags` crate:

```
sigma.rs:        b"t2-commit", b"t2-commit-ch"
adapter.rs:      b"cyclo-ajtai-d2/v1"
greyhound_pcs.rs: b"greyhound-A", b"greyhound-B", b"greyhound-D"
```

While the `domain-tags` crate has 24 registered tags, these inline strings are not validated by CI. If two developers independently choose the same label string (e.g., `b"commit"`) for different protocol contexts, cross-protocol hash collisions become possible.

**Impact**

If domain tags collide, a proof commitment from one protocol (e.g., sigma transcript commitment) could be accepted as valid in another protocol (e.g., Cyclo accumulator commitment), enabling cross-protocol substitution attacks.

**Fix**

1. Move ALL inline domain strings into `domain-tags` as enum variants
2. Add CI lint: `grep -r 'b"' crates/ | grep -v domain_tags` should be empty
3. Add a build-time collision check in the `domain-tags` crate

### M9: FS Transcript Lacks Per-Round Domain Separation in Challenge Expansion · MEDIUM

**File**: `crates/pvthfhe-nizk/src/fiat_shamir.rs:92-111`

**Vulnerability**

The `challenge_bytes` function's counter-mode expansion:

```rust
pub fn challenge_bytes(&mut self, label: &[u8], out: &mut [u8]) {
    // ... absorbs label into self.hasher ...
    let state: [u8; 32] = self.hasher.clone().finalize().into();
    let mut counter: u64 = 0;
    while written < out.len() {
        let mut h = Sha256::new();
        h.update(counter.to_be_bytes());
        h.update(state);         // state is fixed per call
        let block: [u8; 32] = h.finalize().into();
        // ... copy block to output ...
        counter += 1;
    }
}
```

The counter-mode expansion uses `SHA256(counter || state_hash)` where `state_hash` is the label-absorbed transcript hash. This is standard, BUT the counter is incremented independently WITHOUT per-round domain separators. If two different challenge labels produce the same transcript state (collision), the entire challenge byte stream would be identical.

This is the standard counter-mode construction and is cryptographically sound given SHA-256 collision resistance. However, adding label binding to each expansion block (`h.update(label)` before `h.update(counter)`) would provide defense-in-depth.

**Fix**

Add label binding to each expansion block for defense-in-depth:
```rust
let mut h = Sha256::new();
h.update(label);
h.update(counter.to_be_bytes());
h.update(state);
```

### M10: Cyclo Accumulator Metadata Not Bound into Challenge Hash · MEDIUM

**File**: `crates/pvthfhe-cyclo/src/fiat_shamir.rs:7-23`

**Vulnerability**

```rust
pub fn challenge_v1(
    session_id: &str,
    fold_depth: u32,
    acc_commitment: &[u8],
    inst_ajtai_bytes: &[u8],
    inst_public_io_bytes: &[u8],
) -> [u8; 32] { ... }
```

The `challenge_v1` function binds `session_id`, `fold_depth`, `acc_commitment`, and instance data. However, it does NOT bind:
- `params_digest` (Cyclo parameter hash)
- `norm_bound_current` (current accumulator norm)
- `participant_id` (prover identity)

Without `participant_id` binding, a valid challenge for one prover could be replayed for another prover within the same session. Without `params_digest`, the challenge does not commit to the Cypo parameter set.

**Impact**

Cross-prover challenge replay within a session. An adversary who observes a valid fold challenge for party A could reuse it for party B if both parties have compatible instance data.

**Fix**

Add `participant_id` and `params_digest` to the challenge hash:
```rust
pub fn challenge_v1(
    session_id: &str, participant_id: u16, params_digest: &[u8; 32],
    fold_depth: u32, acc_commitment: &[u8],
    inst_ajtai_bytes: &[u8], inst_public_io_bytes: &[u8],
) -> [u8; 32]
```

**File**: `crates/pvthfhe-nizk/src/sigma.rs:85-118`

```rust
let inv_sqrt_m = (3.0 / (m as f64)).sqrt();  // floating-point
```

## Cross-Cutting Observations

### End-to-End Proof Pipeline Analysis

Tracing the full proof pipeline from share generation to on-chain verification:

```
Party P_i: sigma::prove_multi(session_id, pid, c_rns, d_rns, s_i, e_i, d_commitment)
  → generates SigmaMultiProof with k=90 rounds
  → each round: derive_transcript_commitment(t, c, d) → derive_challenge_from_commitment(com, sid, pid, round, d_com) → Poseidon → uniform_ternary
  → serialised as proof bytes (version + ccs_instance_id + ajtai_commitment + sha256_binding + sigma_proof_bytes + cyclo_accumulator_bytes)

Native Verifier: adapter::verify(stmt, proof_bytes)
  → parse proof bytes (version check, ccs_instance_id cross-check, length validation)
  → verify_ajtai_commitment (structural: all-zeros check, range check, element count)
  → verify_accumulator_transcript (session_id match, params_digest match, norm_bound, participant membership)
  → sigma::verify_multi(sigma_binding, pid, sigma_stmt, sigma_multi, d_commitment)
    → for each round: re-derive challenge, check equation c*z_s + z_e = t + ch*d_i
  → hash_bridge::verify(pvss_commitment)

Aggregator: CycloAdapter::fold_all(instances, session_id, rng)
  → for each instance: fold_one into accumulator
  → Cyclo fold: uses challenge_v1 (session_id + fold_depth + commitment + instance data)
  → produces CycloAccumulator

Compressor: ProofCompressor::prove(acc, public_inputs)
  → Nova IVC: CycloFoldStepCircuit folds (commitment_hash, norm, fold_count) via SHA-256
  → produces RecursiveSNARK

On-chain: PvtFheVerifier.sol
  → parse IvcBinding (11 fields, all non-zero checks)
  → verify IVC decider (currently fail-closed — ivcDeciderVerifier = address(0))
  → verify C7 via UltraHonk verifier (aggregator_final circuit)
```
