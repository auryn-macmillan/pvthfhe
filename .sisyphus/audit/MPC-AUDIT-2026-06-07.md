## Fresh Findings (2026-06-07)

### FF1: Cyclo Fold Uses Demo Witness Instead of Real NIZK Witness (F4 Confirmation) · HIGH

**Vulnerability**

`crates/pvthfhe-aggregator/src/folding/mod.rs:448-450` constructs a Cyclo CCS instance using `demo_zero_witness_bytes()` and `demo_one_by_one_matrix_bytes()` instead of extracting the real CCS witness from the deserialized NIZK proof:

```rust
// mod.rs:448-450
ccs_witness_bytes: CcsWitnessSecret::new(demo_zero_witness_bytes()),
...
ccs_matrix_bytes: ProtocolBytes::from(demo_one_by_one_matrix_bytes()),
```

```rust
// mod.rs:459-464
fn demo_zero_witness_bytes() -> Vec<u8> {
    let mut out = Vec::with_capacity(36);
    out.extend_from_slice(&1u32.to_be_bytes());
    out.extend_from_slice(&[0u8; 32]);
    out
}
```

The real NIZK verification (`verify_full_nizk`, line 404-407) runs independently via `CycloNizkAdapter` but its output is never wired into the Cyclo fold instance. The fold path accepts any blob that passes structural checks — the cryptographic proof that the instance is well-formed is never connected to the fold accumulator.

**Impact**: A malicious aggregator can inject unproven Cyclo instances into the accumulator. The structural checks validate metadata (session_id, participant_id, params) but never verify the underlying NIZK proof witness. This breaks the soundness of the Cyclo LatticeFold+ folding layer because the fold relation is verified against a demo witness, not the real one.

**Fix**: Extract CCS witness bytes from the deserialized NIZK proof (which embeds sigma witness material) and wire them into the `CcsPShareInstance::ccs_witness_bytes` field. The `CcsWitnessSecret` type already exists — the bytes need to come from `witness.nizk_proof.proof_bytes` after extracting the sigma witness segment.

### FF2: Witness Exact-Length Validation Missing (F2/H7 Confirmation) · HIGH

**Vulnerability**

`crates/pvthfhe-nizk/src/adapter.rs:374-381` validates only that the witness is non-empty:

```rust
fn validate_witness(witness: &NizkWitness) -> Result<(), NizkError> {
    if witness.secret_share_poly.is_empty() {
        return Err(NizkError::InvalidInput(
            "secret_share_poly must be non-empty",
        ));
    }
    Ok(())
}
```

Then `pad_or_truncate_to_rlwe_n` (line 507-512) silently pads short witnesses or truncates long ones:

```rust
fn pad_or_truncate_to_rlwe_n(v: &[i64]) -> Vec<i64> {
    let mut out = vec![0i64; rlwe_n()];  // N=8192
    let take = v.len().min(rlwe_n());
    out[..take].copy_from_slice(&v[..take]);
    out
}
```

A prover with a 1024-element witness produces a proof that appears valid for N=8192. The sigma verifier checks `z_s = y_s + ch*s_i` element-wise — if `s_i[j] = 0` for j ≥ 1024, then `z_s[j] = y_s[j]` (just the mask), and the verifier accepts. The Ajtai commitment partially mitigates (A·s_i = commitment must match) but a short s_i padded with zeros still commits correctly since all zero-entries produce zero contributions.

**Impact**: A prover can reuse a short witness (N=1024) as if it were full-length (N=8192), producing valid sigma proofs for a smaller-than-claimed polynomial ring. Combined with the N=8/N=8192 gap, this creates a layered witness-substitution surface.

**Fix**:
```rust
fn validate_witness(witness: &NizkWitness) -> Result<(), NizkError> {
    if witness.secret_share_poly.is_empty() {
        return Err(NizkError::InvalidInput("secret_share_poly must be non-empty"));
    }
    if witness.secret_share_poly.len() != rlwe_n() {
        return Err(NizkError::InvalidInput("secret_share_poly must have exactly N coefficients"));
    }
    if witness.error.len() != rlwe_n() {
        return Err(NizkError::InvalidInput("error must have exactly N coefficients"));
    }
    Ok(())
}
```
Then remove `pad_or_truncate_to_rlwe_n` from the prove path.

### FF3: G-N8 — N=8 Circuit vs N=8192 Production (Structural Soundness Gap) · CRITICAL

**Vulnerability**

The Noir circuit that serves as the on-chain verifier's trust anchor operates on N=8 polynomials while production RLWE uses N=8192:

```noir
// circuits/aggregator_final/src/main.nr
global N: u32 = 8;
```

The mapping from N=8192 to N=8 occurs in untrusted native Rust (`aggregate_decrypt_raw_result_poly` in `full_pipeline.rs`) with **no proof of correctness-preservation**. Since the circuit IS the on-chain decision boundary, the security of the entire system reduces to trusting native code that the projection from N=8192 → N=8 is semantically correct. A malicious aggregator (untrusted by design) can choose N=8 projections that satisfy circuit constraints but correspond to an incorrect N=8192 plaintext.

**Impact**: The structural soundness of on-chain verification. Any verification result may be accepting semantically incorrect decryptions. This is not fixable via sigma protocol improvements — the soundness argument operates in the N=8192 domain while verification occurs in N=8.

**Fix**: Scale Noir circuits to N=8192 (requires Noir `generic_const_exprs` or circuit specialization), OR provide a formal reduction theorem proving N=8 correctness implies N=8192 correctness with bounded probability, OR document as a fundamental research-prototype limitation with hard fail-closed behavior.

### FF4: P4 — On-Chain IVC Decider Verification (Fail-Closed) · CRITICAL

**Vulnerability**

The on-chain verifier (`PvtFheVerifier.sol`) uses an external IVC decider contract through the `IIvcDeciderVerifier` interface, but no real decider exists. The `ivcDeciderVerifier` is initialized to `address(0)`, making the `verifyWithIvc` path revert. While this is correctly fail-closed, it means:

1. The `aggregator_final` circuit proves Lagrange recombination at N=8 but does NOT verify the Nova IVC proof chain
2. The `IvcBinding` struct carries 11 fields (including `ivcProofHash`, `vkHash`, `ppHash`, `z0/ziCommitment`, `ivcSteps`) that are checked for non-zero but never cryptographically verified
3. The `ivcVerifyResult` field is explicitly deprecated (line 536 comment: "caller-supplied verify result is NOT trusted")
4. A real decider is the difference between "we check that someone claims the proof is valid" and "we verify the proof ourselves"

**Impact**: The IVC proof chain — which provides soundness for the Cyclo accumulator folding — is not verified on-chain. The current posture is secure (fail-closed) but provides no IVC security guarantee. If a decider is ever configured without being cryptographically sound, the entire on-chain verification collapses.

**Fix**: Implement a real on-chain Nova/LatticeFold IVC decider verifier (e.g., a Solidity contract that verifies RecursiveSNARK proofs, or a Noir circuit). This is tracked as OPEN PROBLEM P4. Until resolved, keep `ivcDeciderVerifier = address(0)`.

### FF5: Schnorr Module Lacks Proof-of-Possession (H8 Confirmation) · HIGH

**Vulnerability**

`crates/pvthfhe-nizk/src/schnorr.rs` provides Schnorr signature verification but does NOT include proof-of-possession (PoP) for public keys used in DKG. Per the MPC audit checklist:

> **Rogue-Key Attack**: Public-key aggregation needs either commit-before-reveal, proof of possession, or both. Otherwise an adversary chooses its key after seeing honest keys and cancels them.

The DKG Round 1 does have commit-reveal (H2 fix), but the Schnorr NIZK used within DKG for well-formedness does not independently enforce that the prover knows the discrete log of their public key. An adaptive adversary could choose a rogue key after observing honest contributions.

**Impact**: Adaptive key cancellation during DKG. The C5 aggregate-key formation proof partially mitigates this at the aggregate level, but individual Schnorr NIZKs lack the independent PoP that standard practice requires for multi-party key aggregation.

**Fix**: Add a proof-of-possession variant to the Schnorr module: `schnorr::prove_pop(sk, pk) → PoP` and `schnorr::verify_pop(pk, PoP) → bool`. Integrate into DKG Round 1: each party must submit PoP before their key is accepted into the aggregate.

### FF6: fiat_shamir::challenge_bytes Lacks Per-Round Label Binding (M9 Confirmation) · MEDIUM

**Vulnerability**

`crates/pvthfhe-nizk/src/fiat_shamir.rs:92-111` counter-mode expansion:

```rust
while written < out.len() {
    let mut h = Sha256::new();
    h.update(counter.to_be_bytes());
    h.update(state);         // label NOT re-bound per block
    let block: [u8; 32] = h.finalize().into();
    ...
}
```

The label is absorbed into the persistent hasher once (line 94-98) but each counter-mode block only hashes `(counter || state_hash)`, not `(label || counter || state_hash)`. While cryptographically sound given SHA-256 collision resistance (different counters produce different inputs), adding label binding per block provides defense-in-depth against transcript collision between different challenge labels producing the same state.

**Fix**:
```rust
let mut h = Sha256::new();
h.update(label);
h.update(counter.to_be_bytes());
h.update(state);
```

### FF7: Poseidon Hash Panics on Invalid Arity (F10 Confirmation) · MEDIUM

**Vulnerability**

`crates/pvthfhe-nizk/src/sigma.rs:725`:

```rust
let mut hasher = Poseidon::<Fr>::new_circom(inputs.len())
    .unwrap_or_else(|_| panic!("Poseidon arity out of circom range: {}", inputs.len()));
```

If `inputs.len()` exceeds the Poseidon implementation's limits, the thread panics. In a production multi-party setting, this causes a party to abort without sending an identifiable error to peers — violating the abort-with-public-blame model.

**Fix**: Return `NizkError::VerificationFailed("Poseidon arity error")` instead of panicking.

### FF9: CycloParams Not Send+Sync (F11 Confirmation) · LOW

**Vulnerability**

`crates/pvthfhe-cyclo/src/lib.rs` — `PVTHFHE_CYCLO_PARAMS` uses `lazy_static!` but the struct is not explicitly `Send + Sync`. In concurrent verification across threads, compilation may succeed but runtime safety is not guaranteed.

### FF10: Lazer Bridge Discards Session/Party Binding · MEDIUM

**Vulnerability**

`crates/pvthfhe-nizk/src/lazer_bridge.rs:275-276` — the LaZer C library bridge:

```rust
pub fn prove(..., _session_id: &str, _participant_id: u16, ...) -> Result<...> {
    let _ = _session_id;    // DISCARDED
    let _ = _participant_id; // DISCARDED
```

When `enable-lazer` is active (the default sigma backend per ARCHITECTURE.md), LaZer-generated proofs do NOT bind to session or participant identity at the Rust/FFI boundary. The binding depends entirely on the C library's internal handling — which is unverified from the Rust side.

**Impact**: If LaZer is the active sigma backend (as documented), proofs are not bound to session or participant identity. A proof from party A in session X can be replayed as party B's proof in session X, or vice versa.

**Fix**: Pass `session_id` and `participant_id` to the LaZer C library's `relation_vk` or `proof` construction. If the C API does not support this, require session/participant binding in the Rust layer before calling LaZer (e.g., by hashing session_id+participant_id into the witness commitment).
