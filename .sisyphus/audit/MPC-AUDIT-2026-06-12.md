## Detailed Findings

### F1 : NonEquiv Schnorr Signatures Lack Session Binding — HIGH

**Vulnerability**

The NonEquiv protocol signs a hash of `(dealer_id, round1_payload)` but the Schnorr challenge derivation has NO `session_id` anywhere in the chain:

```text
hash_round1_message(dealer_id, payload) → SHA256(DOMAIN ":msg-hash:" dealer_id payload)
  → non_equiv_challenge(message_hash)    → SHA256(DOMAIN ":challenge:" message_hash) → Fr
    → schnorr_sign(sk, msg_fr, rng)      → hash_to_challenge(R, PK, msg_fr)
```

```rust
// crates/pvthfhe-non-equiv/src/lib.rs:124-131
pub fn hash_round1_message(dealer_id: u32, round1_payload: &[u8]) -> [u8; 32] {
    let mut h = Sha256::new();
    h.update(DOMAIN_SEPARATOR);           // "pvthfhe-non-equiv/v1"
    h.update(b":msg-hash:");
    h.update(&dealer_id.to_be_bytes());
    h.update(round1_payload);             // NO session_id
    h.finalize().into()
}
```

```rust
// crates/pvthfhe-nizk/src/schnorr.rs:108-119
fn hash_to_challenge(r: &G1Affine, pk: &G1Affine, message: Fr) -> Fr {
    let mut h = Sha256::new();
    h.update(Tag::SchnorrChallenge.as_bytes()); // "pvthfhe/schnorr-challenge/v2"
    h.update(affine_to_bytes(r, true));          // R.x
    h.update(affine_to_bytes(r, false));         // R.y
    h.update(affine_to_bytes(pk, true));         // PK.x
    h.update(affine_to_bytes(pk, false));        // PK.y
    h.update(&msg_bytes);                        // NO session_id
    Fr::from_be_bytes_mod_order(&h.finalize())
}
```

The Schnorr PoP also lacks session binding:
```rust
// crates/pvthfhe-nizk/src/schnorr.rs:69-76
fn pop_challenge(pk: G1Affine) -> Fr {
    let sha = Sha256::new()
        .chain_update(Tag::SchnorrPop.as_bytes()) // "pvthfhe-schnorr-pop-v1"
        .chain_update(pk_bytes)                    // NO session_id
        .finalize();
    Fr::from_be_bytes_mod_order(&sha)
}
```

**Impact**

An adversary can replay a valid Schnorr signature (or PoP) from DKG session A into DKG session B, provided the `(dealer_id, round1_payload)` tuple is identical. For NonEquiv specifically, if a dealer broadcasts the same Round 1 message in two sessions, signatures collected in one session are valid in the other. The session binding is only indirect — through the commitment_nonce embedded in `round1_payload` (simulator.rs:602-609) — but this is fragile and not cryptographically enforced at the signature level.

For PoP: A PoP produced for key registration in one session is valid in any other session. If keys are reused across sessions (long-term keys), this is acceptable. If keys are session-scoped, cross-session replay could impersonate a party.

**Fix**

1. Add `session_id` to `hash_round1_message` in `non-equiv/lib.rs`:
```rust
h.update(session_id);
```

2. Add `session_id` to `hash_to_challenge` in `schnorr.rs`:
```rust
h.update(session_id);
```

3. Add `session_id` to `pop_challenge` in `schnorr.rs` (or document that PoP is intentionally session-independent for long-term keys).

### F2 : AVID Merkle Tree Lacks Session Binding — HIGH

**Vulnerability**

The AVID Merkle tree `leaf_hash` binds `(party_id, share_bytes)` but not `session_id`:

```rust
// crates/pvthfhe-pvss/src/avid.rs:55-62
fn leaf_hash(party_id: u32, share_bytes: &[u8]) -> Fr {
    let mut h = Sha256::new();
    h.update(DOMAIN_SEPARATOR);           // "pvthfhe-avid/v1"
    h.update(b":leaf:");
    h.update(&party_id.to_be_bytes());
    h.update(share_bytes);                // NO session_id
    Fr::from_be_bytes_mod_order(&h.finalize())
}
```

**Impact**

If two different DKG sessions use the same AVID dispersal with the same `(party_id, share_bytes)` pair (e.g., identical shares distributed in two sessions), the Merkle inclusion proof from one session is valid in the other. An adversary could substitute shares from a compromised session B into session A without detection, provided the shares are byte-identical.

**Fix**

Add `session_id` to `leaf_hash`:
```rust
h.update(session_id);
```

### F3 : Leader Election Lacks Session Binding — HIGH

**Vulnerability**

`generate_rank` binds `(seed, party_id)` but not `session_id`:

```rust
// crates/pvthfhe-aggregator/src/leader_election.rs:34-49
pub fn generate_rank(seed: &[u8; 32], party_id: u32) -> ProvableRank {
    let mut h = Sha256::new();
    h.update(DOMAIN_SEPARATOR);           // "pvthfhe-leader-election/v1"
    h.update(b":rank:");
    h.update(seed);
    h.update(&party_id.to_be_bytes());    // NO session_id
    let rank: [u8; 32] = h.finalize().into();
    ...
}
```

**Impact**

If the same election seed is used across two sessions, all party ranks are identical. Since the proof embeds `(seed, party_id)`, a rank from session A is cryptographically indistinguishable from the same (seed, party_id) rank in session B. A leader elected in session A could claim leadership in session B by replaying their rank.

**Fix**

Add `session_id` to `generate_rank`:
```rust
h.update(session_id);
```

### F4 : CycloTernaryTranscript::absorb Has No Label/Length Prefix — MEDIUM

**Vulnerability**

The `CycloTernaryTranscript::absorb` hashes data without any label or length prefix:

```rust
// crates/pvthfhe-cyclo/src/fiat_shamir.rs:122-124
pub fn absorb(&mut self, data: &[u8]) {
    self.state.update(data);              // NO label, NO length prefix
}
```

Compare with the well-constructed `Transcript::absorb` in `pvthfhe-nizk/src/fiat_shamir.rs` which uses length-prefixed encoding.

**Impact**

Without labels or length prefixes, different sequences of `absorb` calls can produce the same hash state if an adversary can control data splits. For example, `absorb(b"AB")` followed by `absorb(b"CD")` produces the same state as a single `absorb(b"ABCD")`. This could allow an adversary to manipulate challenge derivation by controlling how data is chunked across absorb calls.

**Fix**

Apply the same length-prefixed encoding as `Transcript::absorb`:
```rust
pub fn absorb(&mut self, label: &[u8], data: &[u8]) {
    self.state.update(&(label.len() as u64).to_be_bytes());
    self.state.update(label);
    self.state.update(&(data.len() as u64).to_be_bytes());
    self.state.update(data);
}
```

### F5 : NIZK Errors Are Opaque (No Party ID) — MEDIUM

**Vulnerability**

`NizkError`, `FoldError`, most `PvssError` variants lack party identifiers:

```rust
// crates/pvthfhe-nizk/src/lib.rs:138-155
pub enum NizkError {
    InvalidInput(&'static str),         // NO party ID
    VerificationFailed(&'static str),   // NO party ID
    ProofGenerationFailed(&'static str),// NO party ID
    CycloError(String),                 // NO party ID
}
```

```rust
// crates/pvthfhe-pvss/src/lib.rs:98-131
pub enum PvssError {
    InvalidShare,                       // NO party ID
    BfvEncryptionProofFailed,           // NO party ID
    D2HashBindingFailed,                // NO party ID
    StatementMismatch,                  // NO party ID
    // Only SmudgeSlotReused carries party_id: u16
    SmudgeSlotReused { party_id: u16, slot_id: u16 },
    ...
}
```

**Impact**

When a NIZK verification fails, the error tells the caller WHAT failed but not WHO caused it. A verifier cannot issue a `BlameProof` against a specific party because the error lacks the offending party's identity. This violates the "abort with public blame" security property (SEC-8 in threat model).

**Fix**

Add `party_id` fields to all error variants in `NizkError`, `PvssError`, `FoldError`, and `DkgError`. At minimum, errors encountered during per-party proof verification must carry the party ID.

### F6 : `.expect()` in NIZK Paths Destroy Blame Context — MEDIUM

**Vulnerability**

Production `.expect()` calls in `nizk_share.rs` panics on internal invariant violations:

```rust
// crates/pvthfhe-pvss/src/nizk_share.rs:1378
let d_commitment = compute_share_d_commitment(stmt);  // uses stmt fields

// crates/pvthfhe-pvss/src/nizk_share.rs:1394
// .expect() in encoding logic
```

**Impact**

If an adversarial share triggers an invariant violation during proof generation, the process panics instead of returning a structured error. The caller cannot issue a `BlameProof` — the panic destroys all context about which party caused the failure.

**Fix**

Replace `.expect()` calls with proper `Result` returns using `PvssError` variants that include party context.

### F7 : No MPC Round Timeouts — MEDIUM

**Vulnerability**

There is no timeout mechanism for MPC protocol rounds (DKG rounds, share verification, decryption aggregation). The only timeout in the codebase is `run_with_timeout()` in `full_pipeline.rs:3592` for Noir/BB subprocesses.

**Impact**

A malicious party can indefinitely stall the protocol by withholding messages (the `WithholdShare` fault type). The protocol blocks forever waiting for missing shares. This violates the liveness property (SEC-8 in threat model) under the synchronous network model with bounded delays.

**Fix**

Add per-round timeout parameters (Δ₁, Δ₂) to the DKG session and decrypt aggregation. On timeout, issue `BlameProof` against non-responsive parties and abort the round.

### F9 : NonEquivSignature::from_bytes Uses G1Affine::new_unchecked — LOW

```rust
// crates/pvthfhe-non-equiv/src/lib.rs:366
let sig_r = G1Affine::new_unchecked(rx, ry);
```

Deserialization does not validate curve membership. However, `schnorr_verify()` calls `is_on_curve()` before the verification equation, so this is caught at verify time. Risk: any code path that reads a `NonEquivProof` from wire and uses it before verification.

### F11 : No Abort-Time Explicit State Reset API — LOW

State cleanup relies entirely on Rust's `Drop` (via `ZeroizeOnDrop`). There is no explicit `fn reset()` or `fn abort_party()` API. While `Drop` eventually cleans up, the state persists in memory until the struct is dropped (potentially delayed via `Arc` sharing).

---

### F12 : No Cross-Instance Abort Propagation — INFO

The protocol is designed for single-process sequential execution. In a real multi-party deployment where each party runs independently, there is no mechanism for one party's abort to trigger cleanup on other parties' instances.

---

### F13 : FHE Wire Types Don't Validate Algebraic Coefficient Domains — INFO

`KeygenShareV1`, `PublicKeyV1`, `DecryptShareV2` in `fhe/src/wire.rs` validate length bounds but not that polynomial coefficient bytes represent valid field elements. Accepted as research prototype limitation.

## Verified Mitigations (Working Correctly)

The following security properties are properly enforced:

1. **NIZK share proofs**: Session binding via `Transcript::new(session_id, participant_id)` + length-prefixed `absorb` + 12 labeled fields in `derive_challenge`. Domain separator validation on both envelope and opened proof. BFV encryption sigma proof (v4). D2 hash binding. ✅

2. **Sigma protocol**: FS challenge binds `(session_id, participant_id, round_index, d_commitment)`. Proper Keccak256 commitment → SHA-256+Poseidon challenge pipeline. Norm bounds enforced on `z_s`, `z_e`. Ternary challenge validation. ✅

3. **BFV sigma**: Length-prefixed session_id and binding_data in FS challenge. ✅

4. **HashBridge**: Length-prefixed session_id, fixed-width participant_id and share encoding. ✅

5. **Ajtai commitment validation**: Exact byte length, not-all-zeros rejection (M7), range check per coefficient, 13 ring elements count check. ✅

6. **Decrypt aggregation**: Party uniqueness via HashSet, ciphertext_hash match, session_id mismatch detection, dkg_root cross-validation, NIZK verification, share cross-validation. ✅

7. **Smudge slot freshness**: `SmudgeSlotRegistry::check_and_record` prevents reuse. ✅

8. **Keygen simulator H2 commit-reveal**: Binds `(party_id, session_id, pk_i_hash, nonce)`. ✅

9. **C5 PoP**: Binds `(party_id, session_id, pk_bytes, nonce)` with key derivation verification. ✅

10. **Schnorr verify**: `is_on_curve()` checks on both `pk` and `sig_r`. ✅
