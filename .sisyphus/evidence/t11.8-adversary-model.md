# PVTHFHE Decentralized Adversary Model

**Document ID:** T11.8  
**Status:** DRAFT (Phase 1 Evidence)  
**Date:** 2026-05-05  

This document enumerates the adversary types, capabilities, and trust boundaries for PVTHFHE (The Interfold). It serves as the foundation for the T15 threat model and T13 re-audit.

## 1. Adversary Classes

### 1.1 Passive Network Adversary (Eavesdropper)
*   **Formal Capability:** Can observe all peer-to-peer and broadcast traffic (DKG transcripts, PVSS ciphertexts, NIZKs, decryption shares, FHE ciphertexts). Cannot modify or drop messages.
*   **Protocol-Layer Target:** DKG, PVSS, FHE Evaluation, Threshold Decryption.
*   **PVTHFHE Defense:** 
    *   **T-IND-CPA Confidentiality:** Under RLWE and Module-LWE hardness, eavesdropping on ciphertexts or DKG transcripts reveals no information about plaintexts (cite `T-IND-CPA` in `.sisyphus/design/security-proofs.md`).
    *   **Statistical Smudging:** Decryption shares are smudged with $\sigma_{\text{smudge}} = 2^{40} \cdot \sigma_{\text{err}}$ to prevent secret-key leakage (cite `SECURITY.md §Smudging`).
*   **Residual Risk:** Traffic analysis (e.g., packet timing) is out of scope.

### 1.2 Active Network Adversary (MitM / Replay)
*   **Formal Capability:** Can modify, drop, or replay any message in transit. Can participate in the protocol as a fake node (if identity is not authenticated).
*   **Protocol-Layer Target:** All layers (Transport, P2P, On-Chain).
*   **PVTHFHE Defense:**
    *   **Authenticated Channels:** Identities are fixed and authenticated (cite `SECURITY.md §Threat Model`).
    *   **On-Chain Replay Protection:** `SessionRegistry.sol` uses `epoch` and `markEpochConsumed` to prevent session/epoch-level replay (cite T4, `SessionRegistry.sol`).
    *   **Fiat-Shamir Domain Separation:** Prevents cross-protocol transcript reuse (cite `A-STRUCT-3` in `.sisyphus/design/assumptions-ledger.md`).
*   **Residual Risk:** DoS via message dropping is addressed by Liveness/Robustness protocols.

### 1.3 Corrupt Minority Committee Member (< t members)
*   **Formal Capability:** Controls up to $t-1$ nodes. Can see internal shares, withhold shares, or submit malformed/griefing shares.
*   **Protocol-Layer Target:** DKG, PVSS, Threshold Decryption.
*   **PVTHFHE Defense:**
    *   **Threshold Secrecy:** Secret key and plaintexts remain hidden if $|S| < t$ (cite `T-IND-CPA`).
    *   **Withholding/Griefing Mitigations:** `aggregate_decrypt` and `aggregate_keygen` deduplicate `party_id` to prevent sybil-griefing (cite T11.6, `.sisyphus/evidence/t11.6-withholding-griefing.md`).
    *   **Blame Protocol:** Dishonest behavior (e.g., forged PVSS shares) is detectable via `BlameProof` (cite `T-ROBUSTNESS`).
*   **Residual Risk:** Attribution of silent withholding (not sending anything) remains an off-chain/future work item (cite T11.6 H1).

### 1.4 Corrupt Majority Committee (≥ t members)
*   **Formal Capability:** Controls $t$ or more nodes. Can reconstruct the secret key and decrypt any message.
*   **Protocol-Layer Target:** Complete System.
*   **PVTHFHE Defense:** **None.** PVTHFHE strictly requires an honest-majority ($t > n/2$) for confidentiality and correctness (cite `A-STRUCT-4`).
*   **Residual Risk:** Systemic compromise.

### 1.5 Malicious On-Chain Prover (Aggregator)
*   **Formal Capability:** Submits the final aggregated proof to the on-chain verifier. May attempt to forge proofs for incorrect decryption results or bypass threshold constraints.
*   **Protocol-Layer Target:** On-Chain Verifier (`PvtFheVerifier.sol`).
*   **PVTHFHE Defense:**
    *   **T-DEC-SOUND:** Proof soundness ensures that only correct decryptions of validly-formed ciphertexts are accepted (cite `T-DEC-SOUND`).
    *   **T-PV-SOUND:** On-chain verifier verifies the SNARK knowledge soundness, relying on KZG binding on BN254 (cite `T-PV-SOUND`).
    *   **Forged-Share Rejection:** Aggregator cannot include malformed shares in the fold without breaking P1/P2 NIZK soundness (cite T5).
*   **Residual Risk:** Soundness is currently conditional on Open Problems P1 and P2 (cite `README.md`).

### 1.6 Side-Channel Adversary (Timing / Oracle)
*   **Formal Capability:** Can measure execution time or observe error-oracle replies (e.g., distinguishing rejection types).
*   **Protocol-Layer Target:** `pvthfhe-keygen`, `pvthfhe-nizk`, `pvthfhe-fhe`.
*   **PVTHFHE Defense:**
    *   **Constant-Time Comparisons:** Critical commitment and challenge comparisons use `subtle::ConstantTimeEq` (cite T11.5, `.sisyphus/evidence/t11.5-side-channel-audit.md`).
    *   **Uniform Error Sentinels:** FHE backend returns uniform errors to prevent oracle-based plaintext/key extraction (cite T11.5).
*   **Residual Risk:** Micro-architectural side channels (e.g., cache timing) are not mitigated in this version.

## 2. Trust Boundary Inventory

### 2.1 Unconditional Trust
*   **Foundry/EVM Integrity:** We trust the underlying blockchain to execute the verifier contract and `SessionRegistry` correctly according to its bytecode.
*   **Cryptographic Primitives:** We assume the correctness of standard implementations of SHA-256, Keccak-256, and BN254 pairing operations.
*   **Trusted Setup (P3):** We trust that the KZG universal SRS was generated honestly and the toxic waste discarded (cite `A-STRUCT-1`).

### 2.2 Cryptographic Verification
*   **Per-Share Well-Formness:** Verified via Cyclo-companion Ajtai NIZKs (P1).
*   **Aggregation Correctness:** Verified via LatticeFold+ over RLWE (P2).
*   **On-Chain Soundness:** Verified via MicroNova-in-UltraHonk (P3).
*   **Threshold Consistency:** Verified via `SessionRegistry` dkgRoot and participant set checks.

### 2.3 Deferred / Future Work
*   **On-Chain Blame Submission:** `SessionRegistry` accepts session registration but currently lacks a `recordBlame` function (cite T11.6 M3).
*   **Formal Composability:** The joint security of sequential composition (P1 $\to$ P2 $\to$ P3) is assumed structurally but not formally proved (cite `A-STRUCT-5`).
*   **Dynamic Committee Rotation:** Superseding old committees is deferred to future governance modules (cite T11.7).

## 3. Open Problems

### 3.1 P1: Lattice NIZK Soundness
Per-share RLWE NIZK knowledge soundness is conditional. A formal joint-extractor proof (T2) for the Cyclo/Ajtai composition is currently a skeleton (cite `docs/security-proofs/lemma9.md`).

### 3.2 P2: LatticeFold+ over RLWE
Soundness of the folding scheme over polynomial rings depends on the **Lemma 9 Invertibility Heuristic**. It is unproven whether the biased ternary challenge difference is always invertible for $X^{256}+1$ (cite `A-LATTICE-4`).

### 3.3 P3: Full On-Chain Soundness
While `P3RealVerifier.sol` has been stripped of its trusted-signer surrogate, the production VK cannot be emitted as a fully BB-generated Solidity verifier due to a Barretenberg format mismatch (cite `SECURITY.md §P3`).

### 3.4 Robustness under Partition
A formal proof of `T-ROBUSTNESS` (Abort-with-Public-Blame) under asynchronous network partitions is not yet complete. Current liveness assumes a synchronous model for keygen/decryption (cite `SECURITY.md §Threat Model`).
