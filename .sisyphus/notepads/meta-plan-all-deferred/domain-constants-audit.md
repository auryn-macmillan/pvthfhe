## 2. Rust Native Domain Usage

### 2.1 `full_pipeline.rs` — Poseidon-based (CORRECT)

**File:** `crates/pvthfhe-cli/src/full_pipeline.rs`

| Function | Domain Tag | Noir Equivalent | Status |

| `bind_8_with_domain_native()` (line 1854) | `Fr::from(6u64)` | `DOMAIN_AGGREGATOR_D_COMMIT = 6` | ✅ MATCH |

These are used in `build_c7_prover_toml()` which generates the Noir `aggregator_final` Prover.toml. The hash functions (`poseidon_hash_native`) use `light_poseidon::Poseidon::new_circom` with 9 inputs, matching the Noir circuit's `poseidon::poseidon::bn254::hash_9`.

**✅ VERIFIED:** The Rust→Noir hash pipeline for `aggregator_final` is consistent.

### 2.2 `witness_gen.rs` — Rolling Digest (LEGACY/STALE)

**File:** `crates/pvthfhe-circuit-tests/src/witness_gen.rs`

This module uses a **different hash function entirely**:
- `rolling_digest_raw()` — linear accumulator with `DIGEST_DOMAIN = 987_654_321`, `DIGEST_BASE = 131`
- NOT Poseidon. NOT domain-separated with numeric tags.

**Impact:** This module generates witnesses for the full-dimension `decrypt_share` circuit (N=8192). However:
1. The `decrypt_share` Noir circuit (N=8 toy version in `circuits/decrypt_share/src/main.nr`) **uses Poseidon**, not a rolling digest.
2. The `noir_decrypt_share` phase is **marked as deferred** in `pvthfhe_e2e.rs` (line 207): "Phase marker only — not implemented. See deferred plans."
3. Any witness generated by `witness_gen.rs` would **fail** the Noir circuit's constraints because the hash function and domain tags don't match.

**⚠️ WARNING:** If `noir_decrypt_share` is ever un-deferred, the witness generation code in `witness_gen.rs` must be rewritten to use Poseidon with the correct domain tags (1-5).

### 2.3 `pvthfhe-domain-tags` — String-based (SEPARATE SYSTEM)

**File:** `crates/pvthfhe-domain-tags/src/lib.rs`

This crate defines string-based domain tags (e.g., `b"pvthfhe/nova/cyclo-fold/v1"`) used for:
- Wire protocol versioning
- NIZK transcript binding
- Cyclo Ajtai commitment tracking
- PVSS batch tracking

These are a **completely separate** domain separation system from the numeric Poseidon domain tags. No numeric consistency is expected between these two systems.

### Why domains 2-5 have no Rust equivalent

No Rust-side computation of domains 2-5 is needed as long as the public inputs are computed correctly.

## 4. Mismatches Found

### 4.1 `witness_gen.rs` uses wrong hash function (LOW SEVERITY — deferred circuit)

The `witness_gen.rs` module in `pvthfhe-circuit-tests` computes decrypt_share witness values using a **rolling digest** (`DIGEST_DOMAIN = 987_654_321`) instead of Poseidon. This is inconsistent with the Noir `decrypt_share` circuit which uses `poseidon::poseidon::bn254::sponge()` and `poseidon::poseidon::bn254::hash_9()`.

**Mitigation factor:** The `noir_decrypt_share` phase is explicitly deferred in the pipeline (`pvthfhe_e2e.rs:207-210`). The witness_gen.rs module appears to be a legacy artifact or placeholder.

**If fixed:** When `noir_decrypt_share` is un-deferred:
1. Replace `rolling_digest_raw()` with `poseidon_hash_native()` in witness_gen.rs
2. Add domain tag constants matching the Noir values:
   - `const DOMAIN_VECTOR_MERKLE: u64 = 1;`
   - `const DOMAIN_STATEMENT_BINDING: u64 = 2;`
   - `const DOMAIN_CHALLENGE_DERIVE: u64 = 3;`
   - `const DOMAIN_CIPHERTEXT_BINDING: u64 = 4;`
   - `const DOMAIN_DKG_BINDING: u64 = 5;`

### 4.2 No mismatch in active pipeline

The `aggregator_final` circuit witness generation in `full_pipeline.rs` correctly uses:
- Domain 1 for `vector_hash_8` (matching `DOMAIN_VECTOR_MERKLE`)
- Domain 6 for `bind_8_with_domain_native` (matching `DOMAIN_AGGREGATOR_D_COMMIT`)
