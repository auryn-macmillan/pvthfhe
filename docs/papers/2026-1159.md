## Abstract
Asynchronous distributed key generation (ADKG) is a fundamental primitive for building threshold
cryptosystems and fault-tolerant distributed protocols in adversarial network settings. A central objective
in this line of work is to achieve ADKG with O(n
## 2
) communication and constant round complexity
under minimal setup assumptions. Recent progress has led to subcubic-communication ADKG protocols
under different trade-offs. Feng and Tang (CRYPTO 2025) presented an ADKG protocol with O(n
## 2

communication and O(1) rounds, but at the cost of a cubic-communication setup phase in which each
party posts a linear-sized public key on a public bulletin board. In contrast, Abraham et al. (PODC
2026) achieved an ADKG protocol with a standard setup phase, where each party posts only a constant-
sized public key, but with O(n
## 2+1/k

These results leave open whether one can simultaneously obtain constant-round complexity and quadratic

communication and O(1) rounds while requiring only a standard setup phase in which each party posts
a constant-sized public key on a public bulletin board. Our protocol is resilient to a strongly adaptive
adversary corrupting up to f < n/3 parties and assumes only random oracles and secure erasures, both
of which are also assumed by prior subcubic-communication ADKG constructions.
## 1  Introduction
Distributed key generation (DKG) [Ped91] allows a set of n parties to jointly generate a public-secret key
pair  (pk, sk) such that the secret key is never reconstructed in full but instead shared among the parties
using a threshold secret sharing scheme.  Cryptographic operations requiring the secret key can then be
performed by a qualified threshold of parties, but not by fewer. DKG is a fundamental primitive in threshold
cryptography, with key applications including threshold encryption and threshold signature schemes, and it is
widely used in Byzantine fault-tolerant (BFT) distributed protocols to reduce both communication and round
complexity [YMR
## +
## 19, GKKS
## +
22]. In many real-world systems, communication delays are unpredictable,
and message delivery cannot be assumed to be synchronous or to occur within known time bounds. This
has motivated a substantial line of work on DKG in the asynchronous network model [KKMS20, AJM
## +
## 21],
referred to as asynchronous DKG (ADKG), where messages may be arbitrarily delayed, but must eventually
be delivered in finite time.  Over the last several years, ADKG has been studied extensively, leading to
protocols that tolerate an optimal number of Byzantine faults, namely up to f < n/3 corrupted parties.

2.2.2  The Approach in [FT24].
We now recall the construction of Feng and Tang.
Silent setup phase. Their work is built on Garg et al.’s silent threshold encryption (STE) [GKPW24]. In a
silent setup, each party posts an O(λn) bit-sized public key on the public bulletin board. After verifying (and
reading) the parties’ linear-sized public keys against the powers-of-tau reference string [DXR24], each party
can aggregate them into a single O(λn)-sized group public key pk, which suffices for the silent-setup threshold
encryption scheme. Feng and Tang extend STE into silent threshold encryption with tag-homomorphism
(STHE), which supports homomorphic aggregation of ciphertexts encrypted under the same tag and tag-
based threshold decryption.
Initial phase. As is common in consensus constructions, Feng and Tang construct a weak coin protocol
and use it for their consensus phase. A common way to construct such a protocol involves every party secret
sharing a random value for every other party. Parties then personally choose a subset of secrets, and then
uses their sum as a random value.

encrypts a fresh random value under the STHE group public key pk and multicasts the O(λ)-sized ciphertext

## 2.4  Our Techniques: Key Escrow
Our Key Escrow protocol is implemented by having each party obtain an aggregated PVSS transcript that
contains a public encryption key, together with sufficient information to reconstruct the secret key with the
help of the network. In our implementation, each transcript has size O(λn).
In order to have every honest party obtain such a transcript, we have two steps: first, at least one honest
party obtains a transcript, and following that, every party obtains a transcript. To implement the first step,
parties call the PVSSExchange protocol of [ABLS25], which incurs O(λ
## 2
n
## 2
) communication costs. In the
second step, each party that holds a transcript disseminates it in a call to a Disperse protocol and holds
a proof that it has been disseminated.  This proof guarantees that a valid aggregated transcript can be
retrieved by any other party. Parties then forward these proofs to the network. Each party then retrieves a
transcript based on the first verifying proof it hears.
At a later stage, parties should be able to retrieve their secret keys in the Key Retrieve protocol. Recall
that PVSS transcripts contain n encrypted Shamir shares of the secret key. During the Key Retrieve protocol,
each party sends each other party an encryption of its Shamir share. Parties then reply with the decrypted
shares, which can then be combined to compute the shared secret key.

## 2.5  Our Techniques: Provable Rank Protocol
In this section, we describe how to replace the VRFs that were used in the intuitive description above with
an interactive analogue called a Provable Rank protocol. This interactive protocol can be constructed using
a plain PKI setup without additional setup assumptions.  Similarly to VRFs, a Provable Rank protocol
allows parties to output independent pseudorandom values, along with proofs that these values are correctly
computed.
As a first attempt, we could recall that parties could call the Key Escrow protocol to generate a public
key with an associated random secret key. Parties can then call Key Retrieve to get that secret key and
use it as a random value. Unfortunately, there is no guarantee that different parties generate different keys
in the Key Escrow protocol, meaning that while each secret key is indeed random, the keys might not be
independent.
We start in the same manner as the Key Escrow protocol by allowing each party to get an aggregated
PVSS transcript with an unknown secret key. Now, instead of retrieving the secret keys, we view each secret
key as a threshold VRF key. Each party i then collaboratively evaluates the threshold VRF on its own index,
with the help of the network. To do so, i sends every other party its encrypted Shamir share contained in
the PVSS transcript. Parties then reply with partial evaluations of the threshold VRF, which i combines to
## 6

obtain a pseudorandom rank and proof. Note that each party evaluates its threshold VRF on its own index,
meaning that even if different parties use the same transcript, the random values are still independent.

## 2.7  Open Questions
All existing works achieving subcubic communication under strong adaptive security require secure erasures.
A major open question is therefore whether this requirement is inherent, or whether the assumption can
be removed while maintaining subcubic communication (including setup costs). As a first step, one could
explore this direction using heavy cryptographic machinery such as indistinguishability Obfuscation (iO) or
witness encryption for all NP.

Verifiable random function. A verifiable random function (VRF) [MVR99] enables the generation of a

key  sk and an epoch number r ∈ N, outputs the updated secret key  sk for the next epoch r + 1. Looking
ahead, we employ the committee-sampling procedure of [ABLS25], which makes use of forward-secure digital
signatures; these can be instantiated using [KO17].
Public key encryption. In a public-key encryption (PKE) scheme [RSA78], a party holds a public-secret
key pair (ek, dk). The public key  ek is made available to everyone, while the secret key  dk is kept private
by its owner. Any party can encrypt a message m under  ek to produce a ciphertext c =  Encrypt(ek,m).
Only the holder of the corresponding secret key dk can decrypt c to recover the message, by computing m =
Decrypt(dk,c). Chosen-plaintext (CPA) security requires that no PPT adversary can distinguish encryptions
of two messages of its choice, i.e., ciphertexts leak no information about the plaintext beyond its length.
Looking ahead, we will use an encryption scheme to allow parties to encrypt their individual raw ranks; this
can be instantiated using [BF01].

with another. Uniqueness prevents equivocation by asserting that for each participant there is at most one
valid (rank,π) pair; no one can produce different “winning tickets” for the same party. Secrecy hides the
outcome: even though the protocol runs interactively, an adversary who sees all messages and proofs cannot
tell an honest participant’s rank from a fresh random element of D. Termination guarantees liveness: if all
honest parties engage, the protocol eventually finishes and yields a rank–proof pair for each of them, so the
collective random draw cannot be indefinitely blocked by malicious behaviour.

## 7  Weak Leader Election

Assuming all parties execute the protocol with the same tag, a Weak Leader Election protocol must satisfy
the following properties in the presence of an adversary corrupting at most f parties:
## •
α-Correctness. With probability at least α, all honest parties that output an index output the same
index ℓ∈ [n], and the party indexed by ℓ was honest at the beginning of the protocol execution.
## •
Termination. If all honest parties participate in Election(tag), then each of them eventually completes
the protocol and outputs an index.
Intuitively, a Weak Leader Election protocol is designed to select a single honest leader among n par-
ticipants in a distributed setting. α-correctness guarantees that, with probability at least α, all honest
participants agree on the same leader and that the selected leader was honest at the start of the protocol.
Termination ensures liveness: if all honest parties participate, the protocol eventually completes and each
honest party outputs a leader, so the election cannot be indefinitely stalled by malicious behavior.

7.4  Putting the Pieces Together: Our ADKG
We obtain our ADKG protocol as follows. First, parties call the PVSSExchange protocol [ABLS25] in order
to obtain a possible valid output from the ADKG protocol for parties to agree on. Recall that at least one
honest party outputs an aggregated PVSS transcript from the protocol. That party provably disperses it,
and sends the proof to all parties. The parties then execute a VABA protocol to agree on which dispersed
aggregated transcript to retrieve. This is done using the consensus protocol of Abraham et al. [ABLS25],
replacing their recursive weak coin generation procedure with our weak leader election (equivalently, weak
coin) protocol. Their VABA protocol proceeds in consecutive views, which can be thought of as attempts to
reach consensus.
In each view, parties call the weak coin protocol once, and incur O(λn
## 2
) communication and O(1) rounds.
Since our weak coin protocol has constant probability of success, the VABA protocol requires a constant
expected number of views until all parties terminate. Finally, parties can retrieve the agreed upon transcript,
which can be output from the ADKG protocol. The Retrieve protocol requires an additional O(λn
## 2
) cost
and O(1) rounds. Consequently, the overall expected communication cost of our ADKG protocol is O(λ
## 2
n

and it completes in expected O(1) rounds.
On Forward-Secure Signatures. In our protocol, we use forward-secure signatures whose keys are up-
dated during the PVSS Exchange protocol.  This is done to prevent the adversary from learning who is
allowed to send messages to whom and then adaptively corrupting them while they still hold the signing
keys. Parties update their keys in the PVSS Exchange protocol and erase the old key, and only then send
the first message of the protocol. However, to verify signatures generated with a signing key, parties need to
know how many times the key was updated.
Thankfully, at each point throughout the ADKG protocol, parties know exactly how many times the
key was updated. Each call to the election protocol (once per view) starts by calling the Key Escrow and
Provable Rank protocols once, both beginning with a call to the PVSS Exchange protocol. Parties update
their signing key once in each one of these calls. In addition, parties call the PVSS Exchange protocol once
at the beginning of the ADKG protocol, and thus update the key one more time. This means that parties
can know how many times the signing key was updated as a function of the view number, which is attached
to every message in consensus protocols.
On Multi-Instance ADKG. While our construction allows parties to run an ADKG protocol to set up
threshold cryptosystems, parties need to update their secret key from the forward-secure signature scheme
in the process. For many purposes, simply having a single threshold system setup is enough (and it can also
be used as a standard PKI setup for a non-threshold signature scheme). However, if one wishes for parties
to keep their original secret signing key without erasing it after an update, e.g., for running many instances
of ADKG, one can add another round before executing each ADKG instance. Parties start by generating
## 23

another key pair for a forward-secure signature scheme and signing the public key. This signing key pair
will be used as an ephemeral key pair for the ADKG instance. Crucially, parties can update this ephemeral
secret key without changing their permanent signing key. Parties then execute a Non-Equivocation round
(Algorithm 7), inputting their ephemeral public key, to generate a proof that this key is the unique key
used in the given ADKG instance. To allow verification of signatures using this key, parties also send their
ephemeral public key and a proof of its uniqueness whenever signing or forwarding a signature.
