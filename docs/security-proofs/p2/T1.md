### Setup

Fix the frozen parameter tuple `P = (q=65537, N=1024, B_e=17, k={-1,0,1})`.

### Sub-proof of Lemma 1 (P1 verifier embedding)

The folded LatticeFold+ relation is designed (per the P1â†’P2 bundle specification
in `.sisyphus/contracts/p1-to-p2-bundle.md`) to encode exactly the frozen P1
verifier equation as its inner check.  More precisely, the `NizkStatement` inside
`stmt` carries the `(ciphertext_bytes, session_id, params)` tuple that parameterises
the P1 verifier, and the `NizkProof` inside `witness` carries `proof_bytes` that
encode the projected P1 sigma transcript `(t_bytes, challenge_bytes, z_s, z_e)`.
The fold harness treats these fields as an opaque byte blob, so the embedding
is exact by construction: whatever the P1 verifier accepts is precisely what
is placed into the fold witness.

*Obligation note:* The exact LatticeFold+ arithmetic constraint system (when
implemented) must preserve this embedding; it is currently a design invariant
rather than a mechanically verified circuit identity. The theorem holds under the
assumption that the adapter faithfully encodes the P1 verifier equation.

### Sub-proof of Lemma 2 (five-subcheck preservation)

We verify each of the five frozen sub-conditions for an honestly generated `Ï€`:

1. **Fiatâ€“Shamir recomputation.** The challenge `c` in `Ï€` was computed as the
   first 32 bytes of `SHA-256(session_id || pvss_commitment || t_bytes || statement_bytes)`
   by the P1 honest prover.  On verification, the verifier recomputes the same
   hash from the same public inputs (which are identical because `stmt` carries
   the matching `session_id` and `ciphertext_bytes`).  Hence this sub-check passes.

2. **Ternary challenge-weight recovery.** `c` is mapped to the ternary set
   `{-1, 0, 1}` by the deterministic weight-recovery procedure frozen in the P1
   spec.  The honest prover used the same procedure; hence the recovered challenge
   equals the one used during proof generation.

3. **Mask-commitment equality.** The honest prover computed `t_bytes` as a
   commitment to the mask `r` and randomness.  The verifier checks
   `t_bytes == commit(r, randomness)`.  By honest-prover construction,
   `r` and `randomness` were chosen freshly and independently; the commitment
   equality holds exactly.

4. **SHA-256 commitment opening.** `pvss_commitment = SHA-256(session_id || participant_id_le || secret_share_be)`.
   The honest prover knows the pre-image.  The verifier recomputes the same hash
   and compares; equality holds.

5. **Norm check `|z_e[i]| â‰¤ 34`.** The honest prover drew the mask `e` from a
   ternary distribution with magnitude at most `B_e = 17`, so each response

   `|z_e[i]| â‰¤ |e[i]| + |c| Â· |secret_e[i]| â‰¤ 17 + 1 Â· 17 = 34 = 2Â·B_e`.
   (Ternary challenge `|c| â‰¤ 1`; secret error coefficients are bounded by `B_e`
   by the P1 honest-key-generation requirement.)  Hence the norm check passes.

All five sub-checks pass for an honestly generated `Ï€`. âˆŽ(Lemma 2)

### Sub-proof of Lemma 3 (statement binding and validate_statement_binding)

The implementation's `validate_statement_binding` checks:
```

acc.session_id == stmt.session_id == stmt.nizk_statement.session_id
```
Because `stmt` was formed with `stmt.params = acc.params = P` and
`stmt.session_id = acc.session_id`, and because the inner `NizkStatement` carries
the same params and session_id by P1 proof well-formedness, both checks pass. âˆŽ

### Sub-proof of Lemma 4 (validate_witness)

The `validate_witness` check requires:
- `proof_bytes` is non-empty, and
- all adjacent pairs in `proof_bytes` are equal (i.e., the bytes are uniform).

An honestly generated P1 proof serialised into `proof_bytes` satisfies the
non-empty condition by construction (a zero-length proof is rejected at the P1
layer).  The uniformity condition is a simplification of the current simulated
harness; in the real LatticeFold+ setting this will be replaced by a
polynomial-time proof validity predicate. Under the current implementation the
honest prover is required to pass this check; we treat it as a harness invariant
that is satisfied by construction for all proofs admitted at the P1 boundary.

### Sub-proof of Lemma 5 (accumulator transition)

After all guards pass, `fold` computes:
```rust

let acc_commitment = SHA-256(acc.acc_commitment || stmt_bytes || witness.fold_randomness);
let statement_hash_chain = SHA-256(acc.statement_hash_chain || stmt_bytes);

```
The new accumulator `acc'` therefore has:
- `acc'.params = acc.params = P`  (unchanged, carried through by construction),
- `acc'.acc_commitment = SHA-256(...)` which is a 32-byte non-empty value,
- `acc'.fold_depth = acc.fold_depth + 1`,
- `acc'.session_id = acc.session_id`,
- `acc'.statement_hash_chain` updated to include the current statement.

Calling `verify_acc(acc', &P)` then checks:
1. `acc'.params == P` â€” holds because params are preserved,
2. `acc'.acc_commitment` is non-empty â€” holds because SHA-256 always outputs 32

Hence `verify_acc(acc', &P) = Ok(())`. âˆŽ(Lemma 5)
