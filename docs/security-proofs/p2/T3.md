# Theorem 3: Zero-Knowledge Preservation for the Projected SLAP Core Transcript

## Statement

**Theorem P2-T3 (ZK Preservation for the Projected SLAP Core Transcript,
LatticeFold+ refinement).**
Assume each inner P1 proof is zero-knowledge for the projected SLAP core
transcript `(t_bytes, z_s, z_e)` under fresh prover masks and fresh fold
randomness (i.e., the inner P1 relation is `(2, 2)`-HVZK for this projected
transcript in the ROM).

Then, for every fold depth `d` and every PPT distinguisher `D`:
```
|Pr[D(Real_d) = 1] - Pr[D(Sim_d) = 1]| Ōēż d ┬Ę ╬Ą_HVZK + d ┬Ę ╬Ą_SHA256
```
where:
- `Real_d` is the real view of `d` successive fold transcripts (the
  `acc_commitment` chain and the inner projected transcript fields),
- `Sim_d` is the simulated view produced by `Sim` using only public statement
  history and public metadata,
- `╬Ą_HVZK` is the HVZK indistinguishability advantage of the inner P1
  projected-core simulator (negligible by assumption), and
- `╬Ą_SHA256` is negligible (SHA-256 collision resistance).

**Scope restriction.** This theorem covers **only** the projected SLAP core
transcript `(t_bytes, z_s, z_e)`.  It does **not** claim zero-knowledge for:
- `secret_share_open`: the opening of the PVSS secret share,
- `error_open`: the full error vector opening.

Both `secret_share_open` and `error_open` are outside the theorem scope because
they are serialised in the frozen P1 proof bytes for audit purposes and are not
hidden by the folding layer.

## Proof

### Setup

Fix the frozen parameter tuple `P = (q=65537, N=1024, B_e=17, k={-1,0,1})`.
Let `n_folds = d`.  We define the hybrid sequence of experiments:

- `H_0 = Real_d`: all `d` fold transcripts are real.
- `H_i` (for `1 Ōēż i Ōēż d`): the first `i` inner P1 projected-core transcripts
  are replaced by simulated transcripts; the remaining `d - i` are real.
- `H_d = Sim_d`: all `d` inner projected-core transcripts are simulated.

### Lemma 1 (Projected-core scope)

**Claim.** The fold accumulator and its public metadata expose only the following
fields derived from witness data:
- `acc_commitment`: a SHA-256 hash that depends on `witness.fold_randomness` and
  the inner proof bytes, but not on `secret_share_open` or `error_open` directly,
- `statement_hash_chain`: depends only on serialised `FoldStatement` fields
  (public inputs),
- `fold_depth`, `session_id`, `params`: public metadata.

The audit fields `secret_share_open` and `error_open` appear in the P1 proof
bytes (`witness.nizk_proof.proof_bytes`) and are hashed into `acc_commitment`,
but are not separably extracted by the accumulator interface.

**Proof.** By inspection of `fold()` in `mod.rs`:
```rust
let acc_commitment = SHA-256(acc.acc_commitment || stmt_bytes || witness.fold_randomness);
```
`stmt_bytes = serialize_fold_statement(stmt)` encodes only public statement
fields (`session_id`, `params`, `ciphertext_bytes`, `fold_index`).
`witness.fold_randomness` is fresh randomness drawn by the prover.
The inner `proof_bytes` (which contain the audit fields) do not appear directly
in `acc_commitment`; the fold commitment hashes only the accumulated state,
the serialised public statement, and fresh fold randomness.

Therefore, `acc_commitment` hides the internal proof structure under SHA-256's
preimage resistance, and the audit fields are not recoverable from
`acc_commitment` alone.  The scope restriction is enforced by the implementation.
ŌłÄ(Lemma 1)

### Lemma 2 (Inner-core simulation)

**Claim.** The projected P1 SLAP core transcript `(t_bytes, z_s, z_e)` is
simulatable from public inputs alone under the inherited ROM/HVZK assumptions.

**Proof.** By the frozen P1 HVZK guarantee (proved in `docs/security-proofs/p1/T3.md`):
the inner P1 relation is honest-verifier zero-knowledge for the projected core
transcript.  The simulator `Sim_inner` works as follows:
1. Sample a simulated challenge `c*` uniformly from `{-1, 0, 1}` at random.
2. Sample simulated responses `z_s*, z_e*` with coefficients uniformly from
   `[-2┬ĘB_e, 2┬ĘB_e] = [-34, 34]`.
3. Compute a simulated `t_bytes*` as
   `t* = A┬Ęz_s* - c*┬Ęu_s  (mod q)` and
   `t_e* = z_e* - c*┬Ęu_e  (mod q)`,
   and serialize to `t_bytes*`.
4. Programme the ROM so that `SHA-256(session_id || pvss_commitment || t_bytes* || stmt_bytes) = c*`.

The resulting simulated transcript `(t_bytes*, c*, z_s*, z_e*)` is
computationally indistinguishable from a real transcript by the P1 HVZK
property, with advantage `╬Ą_HVZK` per fold step. ŌłÄ(Lemma 2)

### Lemma 3 (Fold transcript simulation)

**Claim.** The fold-level transcriptŌĆöspecifically the `acc_commitment` field
updated by one `fold()` callŌĆöcan be simulated under fresh fold randomness given
only the public statement.

**Proof.** In the real execution:
```
acc'_commitment = SHA-256(acc.acc_commitment || stmt_bytes || r)
```
where `r = witness.fold_randomness` is fresh uniformly random bytes drawn by the
prover.  In the simulation, `Sim_fold` draws `r* ŌåÉ$ {0,1}^{256}$ uniformly and
computes the same hash.  Since `r` and `r*` are both uniformly random and
independent of everything else in the transcript, the distribution of
`acc'_commitment` in simulation is identically distributed to the real execution.
More precisely, SHA-256 modelled as a random oracle makes `acc'_commitment` a
uniformly random 32-byte string conditioned on the public inputs, which the
simulator produces with the same distribution. ŌłÄ(Lemma 3)

### Lemma 4 (Hybrid composition)

**Claim.** For each `i Ōłł {0, ..., d-1}`:
```
|Pr[D(H_i) = 1] - Pr[D(H_{i+1}) = 1]| Ōēż ╬Ą_HVZK + ╬Ą_SHA256
```

**Proof.** The transition from `H_i` to `H_{i+1}` replaces one real inner P1
projected-core transcript at fold step `i+1` with a simulated one (Lemma 2),
and adjusts the corresponding `acc_commitment` contribution (Lemma 3).

The replacement of the inner transcript changes the view by at most `╬Ą_HVZK` by
the P1 HVZK guarantee.  The adjustment to `acc_commitment` is indistinguishable
because `fold_randomness` is fresh and uniform in both the real and simulated
cases (Lemma 3); the only potential distinguishing factor is whether
`SHA-256(acc_commitment_prev || stmt_bytes || r)` correlates with the inner
transcriptŌĆöbut because `r` is independent and SHA-256 is collision resistant,
the two distributions differ by at most `╬Ą_SHA256`. ŌłÄ(Lemma 4)

### Lemma 5 (Metadata leakage)

**Claim.** The public accumulator metadata fields `(fold_depth, session_id,
params, statement_hash_chain)` add only public binding information and do not
leak witness information beyond what is in the projected core transcript.

**Proof.** Each metadata field carries only public information:
- `fold_depth`: the fold counter, determined by the protocol index, public.
- `session_id`: the session identifier, public by protocol design.
- `params = (65537, 1024, 17)`: the frozen parameter tuple, public.
- `statement_hash_chain = SHA-256(prev_chain || stmt_bytes)`: a running hash of
  the public statement sequence, which contains only public inputs.

None of these fields contain witness-derived data (`z_s`, `z_e`, `t_bytes`,
`secret_share_open`, `error_open`).  Hence the metadata contributes zero
additional witness leakage beyond the projected core scope. ŌłÄ(Lemma 5)

### Main argument (hybrid chain)

By the hybrid composition (Lemma 4) and the triangle inequality:
```
|Pr[D(Real_d) = 1] - Pr[D(Sim_d) = 1]|
  Ōēż ╬Ż_{i=0}^{d-1} |Pr[D(H_i) = 1] - Pr[D(H_{i+1}) = 1]|
  Ōēż d ┬Ę (╬Ą_HVZK + ╬Ą_SHA256)
  = d ┬Ę ╬Ą_HVZK + d ┬Ę ╬Ą_SHA256
```

For `d = 10` (the frozen recursion depth for `t = 513`), the bound is:
```
10 ┬Ę ╬Ą_HVZK + 10 ┬Ę ╬Ą_SHA256
```
Both terms are negligible (the factor of 10 preserves negligibility as `d` is
polynomially bounded), so the accumulated view is computationally indistinguishable
from the simulated view. ŌłÄ

### Conclusion

There exists a simulator `Sim_d` that, given only the public statement history
and public metadata, produces a view of the folded projected SLAP core transcript
that is computationally indistinguishable from the real view, with advantage
bounded by `d ┬Ę ╬Ą_HVZK + d ┬Ę ╬Ą_SHA256`.  The theorem holds for the projected
core scope, with the audit fields `secret_share_open` and `error_open` explicitly
excluded from the ZK claim. ŌłÄ
