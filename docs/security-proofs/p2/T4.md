# Theorem 4: Parameter Binding and Accumulator Binding under RingSIS/M-SIS

## Statement

**Theorem P2-T4 (Parameter Binding and Accumulator Binding, LatticeFold+ refinement).**

This theorem has two parts:

**Part A — Parameter binding (unconditional, proved).**
Let `acc` be any accumulator accepted by `verify_acc` at parameter tuple
`P = (q, N, B_e) = (65537, 1024, 17)`.  Then the frozen parameter tuple `P`
is bound throughout the fold transcript: no PPT adversary can produce an
accumulator `acc*` accepted by `verify_acc` with `acc*.params ≠ P` at any fold
step, except with probability `ε_SHA256` (SHA-256 collision resistance).

**Part B — Norm-bound enforcement (conditional security obligation).**
The security of the accumulator commitment against short-vector collisions is
**conditional** on arithmetic enforcement of the norm bound `B_e = 17`:

> *IF* the fold harness enforces `|w[i]| ≤ B_e = 17` on each coefficient of
> every accumulated RLWE witness (i.e., an arithmetic norm check is present),
> *THEN* no PPT adversary can produce two distinct short valid openings
> `w ≠ w'` of the same accumulator commitment `AccCommit(w) = AccCommit(w')`,
> except with probability bounded by the adversary's M-SIS advantage at
> `(q=65537, N=1024, β=2·B_e=34)`.

**Security obligation notice.** The current implementation in
`crates/pvthfhe-aggregator/src/folding/mod.rs` does **not** perform an
arithmetic norm check on the RLWE witness coefficients.  The `validate_witness`
function only checks that `proof_bytes` is non-empty and uniform (a harness
invariant, not a cryptographic norm bound).  Therefore Part B is a
**conditional theorem** and represents an open security obligation: norm-bound
enforcement must be added to the fold harness before the unconditional
accumulator-binding guarantee can be claimed.

## Proof

### Part A: Parameter binding

#### Setup

The frozen parameter tuple is `P = (65537, 1024, 17)` as a Rust tuple
`(u64, usize, u64)`.  The `verify_acc` function checks:
```rust
if acc.params != *expected_params { return Err(FoldError("param mismatch")); }
```
The `fold` function's `validate_statement_binding` checks:
```rust
if acc.params != stmt.params || stmt.params != stmt.nizk_statement.params { ... }
```

#### Lemma A1 (Parameter invariance under fold)

**Claim.** The `params` field of the accumulator is immutable across all fold steps.

**Proof.** In `fold()`:
```rust
Ok(FoldAccumulator {
    params: acc.params,   // copied, not modified
    ...
})
```
The output accumulator carries exactly `acc.params`.  By induction, if the
genesis accumulator has `params = P`, then after any number of fold steps the
accumulated `params` field equals `P`. ∎

#### Lemma A2 (Statement params consistency)

**Claim.** Every statement and inner NIZK statement folded into the accumulator
must carry `params = P`; otherwise `fold()` returns an error.

**Proof.** `validate_statement_binding` enforces
`acc.params == stmt.params == stmt.nizk_statement.params`
before any fold computation proceeds.  An adversary attempting to fold a statement
with `stmt.params ≠ P` will receive `Err(FoldError("param mismatch"))` and the
fold is aborted.  Therefore, any accumulator accepted by `verify_acc` must have
had every folded statement carry `params = P`. ∎

#### Conclusion of Part A

By Lemmas A1 and A2, the parameter tuple `P = (65537, 1024, 17)` is enforced
at every fold step: it cannot be changed without causing `fold()` to return an
error.  Any accumulator accepted by `verify_acc` is therefore bound to `P`.
The probability that an adversary circumvents this is zero (it is a deterministic
equality check, not a probabilistic argument).

The `ε_SHA256` term in the theorem statement arises from the
`statement_hash_chain` binding: the chain records every folded statement's bytes,
which include the `params` encoding.  An adversary who changes `params` in the
chain would need a SHA-256 collision.  But since the params equality check is
deterministic and prior, the `ε_SHA256` term is vacuous for Part A. ∎

### Part B: Norm-bound enforcement (conditional)

#### Why the condition is necessary

The current `validate_witness` in the implementation:
```rust
fn validate_witness(witness: &FoldWitness) -> Result<(), FoldError> {
    let proof_bytes = &witness.nizk_proof.proof_bytes;
    if proof_bytes.is_empty() { return Err(...); }
    if !proof_bytes.windows(2).all(|w| w[0] == w[1]) { return Err(...); }
    Ok(())
}
```
This checks only that `proof_bytes` is non-empty and uniform (all bytes equal).
It does **not** parse the RLWE witness and check that each coefficient satisfies
`|w[i]| ≤ B_e = 17`.  Without such a check, an adversary could submit a
large-norm witness that inflates `z_e` beyond the `34`-bound, potentially
undermining the M-SIS reduction below.

#### Lemma B1 (Commitment linearisation — contingent on LatticeFold+ instantiation)

**Claim (conditional).** *If* the `AccCommit` map used in the full LatticeFold+
accumulator is a linear (or affine) map over `R_q = Z_q[X]/(X^N+1)` with a
publicly sampled matrix `A ∈ R_q^{k×m}`, then `AccCommit(w) = A·w mod q` for
column vectors `w ∈ R_q^m` of short coefficients.

**Proof of conditionality.** In the current hash-chain accumulator, `acc_commitment`
is a SHA-256 hash, not a linear map.  The linear map structure is the design
target for the full LatticeFold+ algebraic commitment.  This lemma is conditional
on the adapter being instantiated with a linear lattice commitment.  Under that
assumption, `AccCommit` is linear by construction.

#### Lemma B2 (Short-opening difference → M-SIS instance)

**Claim (conditional on Lemma B1 and arithmetic norm enforcement).**
*If* an adversary produces two short valid openings `w ≠ w'` with
`AccCommit(w) = AccCommit(w')` and both `|w[i]| ≤ β` and `|w'[i]| ≤ β` for all `i`,
then `Δ = w - w'` is a non-zero short vector with `|Δ[i]| ≤ 2β` satisfying
`A·Δ = 0 mod q`.

**Proof.** By linearity (Lemma B1):
`A·w = A·w' mod q` implies `A·(w - w') = 0 mod q`.
Since `w ≠ w'`, `Δ = w - w' ≠ 0`.
Each coefficient `|Δ[i]| = |w[i] - w'[i]| ≤ |w[i]| + |w'[i]| ≤ β + β = 2β`.
For `β = B_e = 17`, `2β = 34`.
Hence `Δ` is a non-zero short vector in the kernel of `A`, i.e., a solution to
M-SIS(`q=65537`, `N=1024`, `β=34`). ∎

#### Lemma B3 (Hardness of M-SIS at frozen parameters)

**Claim.** M-SIS(`q=65537`, `N=1024`, `β=34`) is computationally hard for PPT
adversaries, providing at least 120-bit post-quantum security under standard
worst-case lattice hardness assumptions.

**Proof sketch.** By the standard M-SIS hardness reduction (Micciancio–Peikert
2012, Lyubashevsky–Peikert–Regev 2010), the hardness of M-SIS at modulus `q`
and degree `N` with norm bound `β` is related to the hardness of approximating
lattice problems (SIVP, GapSVP) on module lattices of rank `N` over `Z_q`.  For
`q = 65537` (a 17-bit prime) and `N = 1024`, the module lattice has dimension
`N = 1024` and modulus `q = 65537`.  The root Hermite factor for breaking
M-SIS with `β = 34` is approximately:
```
δ ≈ (β / q^{1/2})^{1/(N-1)} ≈ (34 / 256)^{1/1023} ≈ 1.00269
```
which corresponds to roughly 120+ bits of post-quantum security by the BKZ
cost model (Albrecht et al., lattice estimator, 2022). ∎

#### Conclusion of Part B (conditional)

*If* (i) arithmetic norm enforcement `|w[i]| ≤ 17` is added to `validate_witness`,
and (ii) the full LatticeFold+ algebraic commitment with a linear map over
`R_q = Z_q[X]/(X^N+1)` is instantiated (replacing the current SHA-256 hash-chain
surrogate),
*then* by Lemmas B1–B3, no PPT adversary can produce two distinct short valid
openings of the same accumulator commitment except with M-SIS advantage, which is
at most `2^{-120}` by standard estimates. ∎

### Security obligation summary

The following implementation changes are required before Part B becomes unconditional:

| Obligation | Description | Status |
|------------|-------------|--------|
| Arithmetic norm check | `validate_witness` must parse RLWE witness and verify `\|w[i]\| ≤ 17` | **IMPLEMENTED** (T20) |
| Linear commitment map | `AccCommit` must be an algebraic linear map over `R_q`, not SHA-256 | **NOT IMPLEMENTED** (design target) |

## Parameter Constraints

| Parameter | Value | Derivation |
|-----------|-------|------------|
| `q`       | 65537 | Frozen NTT-friendly prime; 17-bit modulus |
| `N`       | 1024  | Ring degree; module lattice dimension |
| `B_e`     | 17    | Error bound (NORM OBLIGATION: not yet enforced arithmetically) |
| `β`       | 34    | M-SIS norm bound = `2·B_e` for the difference of two valid openings |
| `k`       | `{-1,0,1}` | Challenge set; bounds response expansion to factor 1 |

The `B_e = 17` parameter is in the frozen tuple and is passed through all fold
steps as `params.2`.  The implementation now evaluates `B_e` as an arithmetic
constraint: `validate_witness` rejects any witness with a coefficient exceeding
`params.2`. (Implemented in T20.)
