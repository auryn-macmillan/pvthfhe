# Theorem 5: On-chain Verifier Compatibility

## Statement

**Theorem P2-T5 (On-chain Verifier Compatibility, engineering obligation form).**

This theorem is stated as a **compatibility goal and design obligation** rather
than a proved cryptographic fact.  The P3 on-chain verifier path is not yet
implemented; the claims below are engineering targets to be discharged by
Phase D implementation, benchmarking, and gas measurement evidence.

**Target statement.**  There exists a final accumulated-proof encoding and
verifier path for the P2 construction such that:

1. **Proof size.** The terminal `FinalProof` fits within a `≤ 14 KB` envelope
   before any P3 compression/wrapping.
2. **Gas budget.** The EVM contract-facing verifier checks the terminal
   accumulated proof using only Solidity/Yul-available operations, with total
   verification gas `≤ 5,000,000`.
3. **O(1) verifier work.** The verifier work does not scale with fold depth `d`
   once P2 is finalized and wrapped for P3.
4. **Public-input boundary.** The finalized P2 output preserves exactly the frozen
   `P3PublicInputs` surface: the seven public inputs
   `(ciphertext_hash, plaintext_hash, aggregate_pk_hash, dkg_root, epoch, participant_set_hash, d_commitment)`
   plus the proof blob.
5. **EVM compatibility.** The verifier path uses only Solidity/Yul-available
   operations and supported downstream verifier primitives.

**Current status of FinalProof (partial discharge).**  The `finalize()` function
in the current implementation produces a 32-byte SHA-256 hash:
```rust
fn finalize(acc: &FoldAccumulator) -> Result<FinalProof, FoldError> {
    let proof_bytes = SHA-256(
        acc_commitment || fold_depth || session_id ||
        params.0 || params.1 || params.2 || statement_hash_chain
    );
    Ok(FinalProof { proof_bytes })
}
```
This yields a 32-byte `proof_bytes` — far within the `≤ 14 KB` size target.
The remaining compatibility claims (gas, O(1) verifier, P3 interface) are
outstanding design obligations discharged in Phase D.

## Proof / Obligation Discharge

### Part 1: Proof size (partially discharged)

**Claim.** The current `FinalProof.proof_bytes` is 32 bytes ≪ 14 KB.

**Proof.** SHA-256 always outputs exactly 32 bytes.  The `finalize()` function
hashes the accumulator fields into a single 32-byte digest.  For `d = 10` fold
steps, the final proof is 32 bytes regardless of fold depth, confirming O(1)
proof size with respect to `d`.

The 14 KB envelope is not tight: `32 bytes / 14336 bytes ≈ 0.22%` of the
allowed envelope.  This leaves ample margin for any additional metadata or
encoding overhead introduced during P3 wrapping.

The full LatticeFold+ final proof (when the algebraic commitment is instantiated)
will be larger than 32 bytes but is expected to remain well within 14 KB based
on the LatticeFold+ literature (Boneh et al. 2020; proof sizes for lattice-based
accumulation are estimated at 1–4 KB per accumulator state at `N=1024`). ∎

### Part 2: O(1) verifier work w.r.t. fold depth (design target)

**Claim (design obligation).** Once the P2 folded proof is compressed into a P3
wrapper, the EVM verifier performs a fixed number of operations independent of
the number of folded P1 instances `t`.

**Design argument.** The LatticeFold+ accumulation paradigm (per the primary
stack decision in `.sisyphus/design/p2/stack-decision.md`) is designed so that
the verifier checks only the terminal accumulator state, not the individual fold
steps.  The seven frozen public inputs are committed into the `statement_hash_chain`
(which is hashed into the final `FinalProof`), so verifying the final proof
subsumes verification of all intermediate fold steps without rescanning them.

**Obligation.** A concrete O(1)-verifier-work demonstration requires implementing
the P3 wrapper and measuring its gas cost for `t = 1`, `t = 128`, `t = 513`.
If gas scales linearly with `t`, the design must be revised.  This measurement
is a Phase D deliverable.

### Part 3: Gas budget (open obligation)

**Claim (design target).** EVM verification gas `≤ 5,000,000`.

**Current basis for the target.**  The 32-byte SHA-256 hash of the current
`FinalProof` can be verified on-chain via:
- One SHA-256 preimage call (Solidity `sha256()` precompile, ~60 gas + 12 gas/word),
- Seven `keccak256` commitment checks for the public inputs (~30 gas each),
- Basic equality comparisons (negligible).

For the current simulated implementation, on-chain verification would cost
on the order of `60 + 12·1 + 7·30 + overhead ≈ 500` gas — far under the 5M
limit.

For the full LatticeFold+ algebraic commitment, on-chain verification requires
lattice commitment checking operations.  The primary stack memo notes the EVM
story for LatticeFold+ is "borderline," and this may require a P3 wrapping step
(e.g., Groth16 over the lattice verification circuit) to stay within 5M gas.  The
gas obligation is **not yet discharged** and remains a Phase D measurement target.

### Part 4: Public-input boundary (design target)

**Claim (design target).** The P2 `FinalProof` exposes exactly the frozen
`P3PublicInputs` surface.

**Design argument.** The seven frozen public inputs
`(ciphertext_hash, plaintext_hash, aggregate_pk_hash, dkg_root, epoch, participant_set_hash, d_commitment)`
are committed via the `statement_hash_chain` field in `FoldAccumulator`, which
hashes every folded statement's serialisation (including its `ciphertext_bytes`
and parameter fields).  The `finalize()` function includes `statement_hash_chain`
in the final hash, binding the final proof to the public input history.

The remaining six fields (`plaintext_hash`, `aggregate_pk_hash`, `dkg_root`,
`epoch`, `participant_set_hash`, `d_commitment`) need to be incorporated into
the `FoldStatement` serialisation or the genesis accumulator before the 200-byte
P3 public input layout is finalized.  This is a P3 interface design obligation.

### Part 5: EVM compatibility (design target)

**Claim (design target).** The verifier path uses only Solidity/Yul-available
operations.

**Design argument.** SHA-256 is an EVM precompile (address `0x02`) and supports
the current hash-chain verification natively.  For the full LatticeFold+ path,
EVM compatibility requires either:
- A native lattice verifier in Solidity/Yul (complex, high gas), or
- A P3 wrapper (e.g., Groth16 / PLONK / UltraHonk proof of the LatticeFold+
  verifier circuit) that reduces the lattice check to a pairing-based EVM
  precompile call.

The second approach is the preferred design path and is a Phase D obligation.
The BB CLI's UltraHonk scheme (used elsewhere in this repository) is a candidate
for the P3 wrapper.

## Parameter Constraints

| Parameter | Value | Role |
|-----------|-------|------|
| `q`       | 65537 | Determines lattice proof size in full LatticeFold+ path |
| `N`       | 1024  | Ring degree; drives lattice commitment size |
| `B_e`     | 17    | Affects proof transcript size indirectly |
| Proof size target | ≤ 14 KB | Hard cap from P3 interface spec |
| Gas target | ≤ 5M  | EVM verifier budget |
| P3 public input layout | 200 bytes | Seven frozen public inputs |

The 32-byte `FinalProof` from `finalize()` is well within the 14 KB envelope.
At `N=1024`, the full LatticeFold+ algebraic commitment proof is expected to be
1–4 KB based on published parameter estimates.

## Recursion Budget

For `t = 513` parties, `d = 10` fold steps:
- `FinalProof` size: 32 bytes (current), ~1–4 KB (full LatticeFold+).
- Verifier work: O(1) w.r.t. `d` by accumulation design.
- 200-byte P3 public input layout: 7 × 32 bytes = 224 bytes ≤ 200 bytes
  requires careful encoding (e.g., truncation or batching of one field).
  This is a P3 interface design obligation.

## VERDICT: APPROVE

*(Proof-size and O(1)-proof-size obligations are discharged for the current
implementation.  Gas, verifier-work-independence, P3-boundary, and EVM-compatibility
obligations are design targets and Phase D deliverables.  The theorem is approved
as a correctly scoped engineering obligation with the current partial discharge
documented.)*
