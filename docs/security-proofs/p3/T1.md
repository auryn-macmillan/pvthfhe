# P3-T1 — Completeness of the ECDSA On-Chain Verifier

**Theorem ID**: P3-T1  
**Status**: PROVED  
**Reduction target**: Direct computation (no hardness assumption required)

---

## Statement

**Theorem P3-T1 (Completeness).** Let `TRUSTED_SIGNER` be the Ethereum address
corresponding to private key `sk` over secp256k1. Let `publicInputs` be any 200-byte
`P3PublicInputs` blob. Let `msg_hash = keccak256(publicInputs)` and let
`(v, r, s)` be an ECDSA signature over secp256k1 produced by the holder of `sk` on the
message `msg_hash` (i.e., `(v, r, s) = ECDSA.Sign(sk, msg_hash)` with canonical
low-`s` form). Then the deployed Solidity on-chain verifier `verify(publicInputs, v, r, s)`
returns `true`.

Formally: for all `sk`, all 200-byte blobs `publicInputs`, and all `(v, r, s)` produced by
`ECDSA.Sign(sk, keccak256(publicInputs))`:

```
ECDSA.Recover(keccak256(publicInputs), v, r, s) = address(pk)  ∧
address(pk) = TRUSTED_SIGNER
⟹ verify(publicInputs, v, r, s) = true
```

---

## Proof

The proof is by direct computation; no hardness assumption is invoked.

**Step 1 — Correct key pair.** The signer holds `sk` such that `address(ECDSA.PubKey(sk)) = TRUSTED_SIGNER`. This is the definitional property of the trusted signer.

**Step 2 — Hash computation.** The verifier computes `msg_hash = keccak256(publicInputs)`. Because `keccak256` is a deterministic function, the verifier and signer compute identical values from the same 200-byte blob.

**Step 3 — ECDSA signature correctness.** By the correctness property of ECDSA over secp256k1 (IEEE P1363 / RFC 6979): for any message hash `m` and any private key `sk`, the algorithm `ECDSA.Sign(sk, m)` produces `(v, r, s)` such that `ECDSA.Recover(m, v, r, s) = address(ECDSA.PubKey(sk))`. This is the standard completeness property of the ECDSA scheme; it holds deterministically (the signing may use RFC 6979 deterministic nonce, or any fresh nonce; recovery succeeds regardless of nonce choice).

**Step 4 — Address comparison.** The verifier calls the EVM `ecrecover` precompile (address `0x01`) with inputs `(msg_hash, v, r, s)`. The `ecrecover` precompile implements `ECDSA.Recover` exactly (Ethereum Yellow Paper §F, Appendix F). It returns the 20-byte address derived from the recovered public key. By Step 3, this equals `address(ECDSA.PubKey(sk)) = TRUSTED_SIGNER`.

**Step 5 — Equality check.** The verifier checks `ecrecover(...) == TRUSTED_SIGNER`. By Step 4 this equality holds, so the verifier returns `true`.

**Conclusion.** The verification always succeeds for honestly produced signatures. ∎

---

## Dependencies

| Dependency | Role |
|-----------|------|
| ECDSA correctness (secp256k1, IEEE P1363 / RFC 6979) | Guarantees `Sign` → `Recover` round-trip |
| EVM `ecrecover` precompile specification (Ethereum Yellow Paper §F) | Guarantees precompile implements `ECDSA.Recover` faithfully |
| `keccak256` determinism | Guarantees identical hash at sign and verify time |

No hardness assumption is required for completeness; the proof is purely algorithmic.
