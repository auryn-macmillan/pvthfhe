# P3-T2 — Soundness of the ECDSA On-Chain Verifier (EUF-CMA Reduction)

**Theorem ID**: P3-T2  
**Status**: PROVED  
**Reduction target**: ECDSA Existential Unforgeability under Chosen-Message Attack (EUF-CMA) on secp256k1

---

## Statement

**Theorem P3-T2 (Soundness / Unforgeability).** Let `TRUSTED_SIGNER` be the Ethereum
address of public key `pk*` over secp256k1. Let `A` be any probabilistic polynomial-time
(PPT) adversary that does not hold the corresponding private key `sk*`. Then for any
200-byte blob `publicInputs`:

```
Pr[ verify(publicInputs, v, r, s) = true  ∧
    (v, r, s) was not produced by the holder of sk* ] ≤ ε_EUF-CMA(κ)
```

where `ε_EUF-CMA(κ)` is the ECDSA EUF-CMA advantage of any PPT adversary against
secp256k1 at security parameter `κ`, which is negligible under the Elliptic Curve
Discrete Logarithm Problem (ECDLP) hardness assumption on secp256k1.

Equivalently: the only way a PPT adversary can make `verify()` return `true` on any
`publicInputs` is by producing a valid ECDSA signature from `sk*`, which requires
solving ECDLP or breaking ECDSA EUF-CMA — both computationally infeasible in the
standard model under the generic group model / ROM assumptions underlying secp256k1
at the 128-bit security level.

---

## Proof

The proof is a straightforward reduction from ECDSA EUF-CMA to the soundness of the
verifier.

**Construction of the reduction.** Suppose for contradiction that there exists a PPT
adversary `A` that, without the private key `sk*`, produces `(publicInputs*, v*, r*, s*)`
such that `verify(publicInputs*, v*, r*, s*) = true` with non-negligible probability `δ`.

We construct a forger `F` against ECDSA EUF-CMA on secp256k1 as follows:

1. **Setup.** `F` receives the ECDSA challenge public key `pk*` and sets
   `TRUSTED_SIGNER := address(pk*)`. `F` runs `A` with this signer address.

2. **Signing oracle simulation.** The EUF-CMA game provides `F` with access to a
   signing oracle `Sign(sk*, ·)`. If `A` ever requests that the signer sign some
   message (in a hypothetical interactive extension), `F` forwards to its signing
   oracle. In our non-interactive setting, `A` does not query a signing oracle;
   `A` must produce a forgery on a fresh `publicInputs*` from scratch. This is
   only stronger (the no-query case is a special case of CMA).

3. **Forgery output.** When `A` outputs `(publicInputs*, v*, r*, s*)` such that
   `verify()` returns `true`, the verifier has confirmed:
   ```
   ecrecover(keccak256(publicInputs*), v*, r*, s*) = TRUSTED_SIGNER = address(pk*)
   ```
   This means `(v*, r*, s*)` is a valid ECDSA signature under `pk*` on the message
   `keccak256(publicInputs*)`.

4. **Reduction conclusion.** `F` outputs `(msg* = keccak256(publicInputs*), v*, r*, s*)`
   as its ECDSA forgery. Since `A` does not hold `sk*`, by assumption `(v*, r*, s*)`
   was not produced by a legitimate signing query. Therefore `F` has produced an
   ECDSA forgery under `pk*` with probability `δ`.

5. **Contradiction.** By the ECDSA EUF-CMA security of secp256k1, `δ` must be
   negligible in `κ`. Therefore `A` cannot exist with non-negligible advantage.

**Tightness.** The reduction is tight: `Adv_A[verify-soundness] ≤ Adv_F[ECDSA-EUF-CMA]`.
There is no loss factor because `F` uses `A`'s output directly as its forgery.

**Note on `keccak256` as message pre-processing.** The message passed to `ecrecover`
is `keccak256(publicInputs)`, not `publicInputs` directly. A pre-processing hash does
not weaken EUF-CMA: if `A` could find a valid signature on some `publicInputs*` whose
hash collides with a hash of a previously signed message, that would require finding a
keccak256 second pre-image. The probability of such a collision is bounded by the
second-preimage resistance of keccak256 (see T5). Hence the total soundness error is:

```
ε_total ≤ ε_EUF-CMA(κ) + ε_SPR-keccak256(κ)
```

Both terms are negligible. ∎
