# P3-T3 — Trusted-Setup Analysis for the ECDSA On-Chain Verifier

**Theorem ID**: P3-T3  
**Status**: PROVED  
**Reduction target**: ECDSA key security (long-term key compromise model); no pairing-based trusted setup

---

## Statement

**Theorem P3-T3 (Trusted-Setup Security — ECDSA Option C).** The P3 on-chain
verifier (Option C) uses ECDSA secp256k1 via `ecrecover` with a hardcoded
`TRUSTED_SIGNER` address. This design has **no pairing-based trusted setup**
(no Powers of Tau, no Groth16 ceremony, no toxic waste). The entire trust
assumption is reducible to the long-term security of the `TRUSTED_SIGNER`
private key `sk*`.

**Claim A (No toxic-waste risk).** Unlike Groth16 or KZG-based verifiers,
the ECDSA verifier requires no structured reference string derived from a
secret trapdoor. There is no ceremony, no toxic waste, and no auxiliary
secret whose disclosure could enable universal proof forgery. Compromising
`sk*` enables forgery of signatures (i.e., forgery of P3 proofs for any
`publicInputs`), but it does not expose any other cryptographic secret and
does not retroactively invalidate past accepted proofs recorded on-chain.

**Claim B (Single-point trust).** The `TRUSTED_SIGNER` address is the sole
trust anchor. An adversary who recovers `sk*` can forge arbitrary valid proofs
for the deployed verifier. The risk model is therefore: the security of the
P3 verifier equals the security of the `TRUSTED_SIGNER` key under the
long-term key compromise threat model.

**Claim C (Key rotation / mitigation).** Key compromise risk is mitigated by
standard key-management practices: hardware security modules (HSMs), threshold
ECDSA for the signer key, offline key storage, and contract-upgrade-based
key rotation. A contract upgrade that changes `TRUSTED_SIGNER` to a new
address immediately invalidates any advantage an adversary gains by
compromising `sk*` going forward (though proofs accepted before the rotation
remain valid on-chain records).

---

## Proof

**Proof of Claim A.** The `verify()` function executes:
```solidity
bytes32 msgHash = keccak256(publicInputs);
address recovered = ecrecover(msgHash, v, r, s);
return recovered == TRUSTED_SIGNER;
```
This computation references no structured reference string, no elliptic-curve
group generators beyond those hardcoded into the secp256k1 standard, and no
ceremony-derived parameters. The `ecrecover` precompile itself (EVM address `0x01`)
is defined by the Ethereum protocol specification and requires no setup beyond the
fixed secp256k1 curve parameters. Therefore, there is no trusted-setup ceremony to
conduct, no toxic waste to discard, and no CRS whose leakage would universally
compromise soundness. ∎ (Claim A)

**Proof of Claim B.** Suppose adversary `A` recovers `sk*` such that
`address(ECDSA.PubKey(sk*)) = TRUSTED_SIGNER`. Then `A` can call
`ECDSA.Sign(sk*, keccak256(publicInputs))` for arbitrary `publicInputs` and
produce `(v, r, s)` that satisfies `verify() = true`. This is exactly an
ECDSA forgery, now trivially achievable because `A` holds the key. Conversely,
by T2, any adversary without `sk*` cannot forge. Therefore, the soundness of
the verifier is exactly equivalent to the confidentiality of `sk*`. ∎ (Claim B)

**Proof of Claim C (informal).** Consider a contract with an `updateSigner(address)`
function protected by governance or admin multi-sig. After key rotation to a new
address `TRUSTED_SIGNER'` derived from fresh key `sk'`:
- New calls to `verify()` compare against `TRUSTED_SIGNER'`; knowledge of the
  old `sk*` is irrelevant.
- Historical on-chain records of accepted proofs are immutable; they remain valid
  evidence of past verifications regardless of key rotation.
- Therefore, key rotation bounds the forward damage from a key compromise event. ∎

**Threat summary.**

| Threat | Consequence | Mitigation |
|--------|------------|------------|
| `sk*` compromised | Arbitrary future proof forgery | Key rotation via contract upgrade |
| Side-channel on `sk*` | Same as above | HSM / threshold ECDSA for signer |
| Ethereum `ecrecover` precompile bug | Soundness broken at EVM level | Out of scope for P3; shared with all Ethereum contracts |
| secp256k1 ECDLP broken | `sk*` extractable from public key | Post-quantum migration required (out of scope) |
