# P3-T5 — Cross-Input Binding of the ECDSA On-Chain Verifier

**Theorem ID**: P3-T5  
**Status**: PROVED  
**Reduction target**: keccak256 second-preimage resistance (SPR)

---

## Statement

**Theorem P3-T5 (Cross-Input Binding).** Let `verify(publicInputs, v, r, s)` be the
deployed Solidity on-chain verifier. Let `publicInputs_A` and `publicInputs_B` be two
distinct 200-byte `P3PublicInputs` blobs (`publicInputs_A ≠ publicInputs_B`). Then
for any `(v, r, s)`:

```
verify(publicInputs_A, v, r, s) = true
⟹ verify(publicInputs_B, v, r, s) = false
```

except with probability at most `ε_SPR(κ)`, the second-preimage advantage of any
PPT adversary against keccak256 at security parameter `κ`, which is negligible
(bounded by `2^{-128}` under the random-oracle assumption for keccak256).

Equivalently, a proof `(v, r, s)` that is valid for one set of public inputs is
cryptographically bound to those exact inputs and cannot be replayed against any
different set of inputs.

---

## Proof

**Step 1 — ECDSA message binding.** The `verify()` function computes:
```solidity
bytes32 msgHash_A = keccak256(publicInputs_A);
address recovered  = ecrecover(msgHash_A, v, r, s);
```
By the correctness of ECDSA over secp256k1, `ecrecover(m, v, r, s)` returns
`address(pk)` if and only if `(v, r, s)` is a valid ECDSA signature on message `m`
under key `pk`. Therefore, `verify(publicInputs_A, v, r, s) = true` implies:

```
(v, r, s) is a valid ECDSA signature on keccak256(publicInputs_A) under sk*
```

**Step 2 — Distinctness implies distinct hashes (reduction from SPR).** Suppose
for contradiction that an adversary `A` finds `publicInputs_B ≠ publicInputs_A` such
that `verify(publicInputs_B, v, r, s) = true` with the same `(v, r, s)`. Then:

```
ecrecover(keccak256(publicInputs_B), v, r, s) = TRUSTED_SIGNER
```

Combined with Step 1, `(v, r, s)` is a valid ECDSA signature on both:
- `m_A = keccak256(publicInputs_A)`, and  
- `m_B = keccak256(publicInputs_B)`.

By the ECDSA signature scheme, `ecrecover` maps each signature to the unique
message it was signed on (given the signature is deterministic in the `(r, s)` pair
for a fixed nonce). For the same `(v, r, s)` to verify against two different
messages `m_A` and `m_B`, we would need `m_A = m_B`, i.e.:

```
keccak256(publicInputs_A) = keccak256(publicInputs_B)
```

with `publicInputs_A ≠ publicInputs_B`. This is exactly a keccak256
**second-preimage**: given `publicInputs_A`, find `publicInputs_B ≠ publicInputs_A`
with the same hash.

**Step 3 — SPR bound.** Under the random-oracle model (ROM), keccak256 is a
256-bit hash function. The second-preimage advantage for any PPT adversary is
bounded by `2^{-128}` (birthday-attack lower bound for a 256-bit hash under ROM;
the exact second-preimage bound is `2^{-256}` for any fixed input since the hash
function is a random oracle). Therefore:

```
Pr[verify(publicInputs_B, v, r, s) = true | verify(publicInputs_A, v, r, s) = true]
  ≤ Pr[keccak256(publicInputs_A) = keccak256(publicInputs_B)]
  ≤ ε_SPR(κ) = 2^{-256}
```

**Step 4 — ECDSA nonce non-collision (tightening).** In more detail, given a fixed
ECDSA signature `(v, r, s)`, the value `r` encodes the x-coordinate of the ephemeral
public key `R = k·G` for signing nonce `k`. The ECDSA verification equation
`u1·G + u2·pk = R` has at most one solution `m` for a given `(v, r, s)` and `pk`
(up to the negligible probability of `r = 0` or `s = 0`, which are excluded by the
signature well-formedness check). Therefore, a single `(v, r, s)` can be a valid
signature on at most one message per public key, making the cross-binding argument
even tighter than the SPR bound alone.

**Conclusion.** A valid proof `(v, r, s)` for `publicInputs_A` is binding to those
exact public inputs. Replaying it against any different `publicInputs_B` succeeds
with probability at most `ε_SPR = 2^{-256}`, which is negligible. ∎

## Dependencies

| Dependency | Role |
|-----------|------|
| keccak256 second-preimage resistance (SPR) | Main binding argument |
| keccak256 as a random oracle (ROM) | Tightness of SPR bound |
| ECDSA message uniqueness (per public key) | Step 4 tightening |
| EVM `ecrecover` precompile faithfulness | Correctness of on-chain hash-and-recover |
